## Context

O produto coleta dados financeiros sensíveis de usuários brasileiros. LGPD (Lei 13.709/2018) é obrigatória desde o dia 1 — não é opcional nem adornamento jurídico.

**O que está faltando para LGPD compliance:**
- Usuário não consegue exportar seus dados (Art. 18 III — portabilidade)
- Usuário não consegue excluir a conta e todos os dados (Art. 18 VI — eliminação)
- Política de Privacidade e Termos de Uso não são acessíveis dentro do app
- Sem tratamento de offline: usuário sem rede vê erro técnico em vez de mensagem clara

**Esta story também trata edge cases críticos** que ficaram fora do escopo dos Epics 1-3:
- Timeout nas chamadas de IA (chat trava indefinidamente sem resposta)
- Mensagem clara quando o limite de perguntas é atingido
- Handling de erros de rede em formulários críticos

---

## User Story

> Como usuário,
> quero ter controle total sobre meus dados financeiros — poder exportá-los ou excluí-los quando quiser,
> e quero que o app se comporte bem mesmo quando estou sem conexão,
> para que eu confie no app com minhas informações financeiras sensíveis.

## Acceptance Criteria

### AC1: Exportação de dados (LGPD Art. 18 III — Portabilidade)

- [ ] Botão "Exportar meus dados" em `/settings` → `?tab=dados`
- [ ] API route `GET /api/user/export` — server-side, requer auth
- [ ] Exporta em JSON estruturado com: perfil, todas as transações (data, descrição, valor, categoria), orçamentos, objetivos, histórico de conversas com IA (`chat_messages`), regras de categorização do usuário (`category_dictionary`)
- [ ] Response: `Content-Disposition: attachment; filename="meus-dados-finansim-YYYY-MM-DD.json"`
- [ ] Rota protegida — só o próprio usuário pode exportar seus dados (verificar `user.id` na query)
- [ ] Loading state no botão durante geração do arquivo
- [ ] **Verificação:** Clicar "Exportar" → download de JSON com todas as transações do usuário

### AC2: Exclusão de conta (LGPD Art. 18 VI — Eliminação)

- [ ] Botão "Excluir minha conta" em `/settings?tab=dados` — posição visível mas não proeminente (evitar clique acidental)
- [ ] Fluxo de confirmação em 2 etapas:
  1. Modal explicando o que será excluído e que é irreversível
  2. Campo de texto: usuário deve digitar "EXCLUIR" para confirmar
- [ ] API route `DELETE /api/user/account` — server-side, requer auth
- [ ] Operação de exclusão em ordem:
  1. Delete de todas as transações do usuário
  2. Delete de budgets, goals, category_dictionary, category_cache (pelo user_id)
  3. Delete da subscription no Asaas (chamar API do Asaas se houver assinatura ativa)
  4. Delete da subscription no banco
  5. Delete do profile
  6. `supabase.auth.admin.deleteUser(userId)` — usando service role
- [ ] Após exclusão: redirecionar para `/login` com query param `?deleted=true`
- [ ] Página de login: se `?deleted=true`, mostrar mensagem "Conta excluída com sucesso. Seus dados foram removidos permanentemente."
- [ ] **Verificação:** Confirmar exclusão → logout + dados removidos + mensagem de confirmação

### AC3: Política de privacidade e termos de uso

- [ ] Página `/privacy` — texto completo da política de privacidade (LGPD compliance)
- [ ] Página `/terms` — texto completo dos termos de uso
- [ ] Ambas acessíveis sem autenticação (rotas públicas fora do grupo `(app)`)
- [ ] Links no rodapé do app (em `/login`, `/signup`) e em `/settings?tab=dados`
- [ ] Conteúdo mínimo da política de privacidade:
  - Quais dados são coletados (dados de conta, transações financeiras)
  - Como são usados (análise pessoal, categorização IA)
  - Com quem são compartilhados (apenas Anthropic/OpenAI para processamento IA)
  - Direitos do titular (exportação, exclusão, retificação)
  - Contato para solicitações LGPD (email de contato)
- [ ] Conteúdo mínimo dos termos: uso permitido, limitações de responsabilidade, condições de cancelamento
- [ ] **Verificação:** Links de privacy/terms levam às páginas corretas; páginas acessíveis sem login

### AC4: Offline handling

- [ ] Hook `useOnlineStatus()` que detecta `navigator.onLine` e escuta eventos `online`/`offline`
- [ ] Banner não-intrusivo no topo do app quando offline: "Você está sem conexão. Os dados exibidos podem estar desatualizados."
- [ ] Banner desaparece automaticamente quando a conexão é restaurada
- [ ] Dashboard e lista de transações continuam exibindo dados do cache React Query quando offline (não quebra, não mostra tela de erro)
- [ ] Formulários que requerem conexão (chat, import) mostram mensagem "Sem conexão. Verifique sua internet." em vez de erro técnico
- [ ] **Verificação:** Desativar rede → banner aparece; dados do dashboard ainda visíveis; chat mostra msg offline

### AC5: Timeout handling nas chamadas de IA

- [ ] `src/hooks/use-chat-timeout.ts` ou lógica inline no `chat/page.tsx`
- [ ] Se o status de streaming ficar `'streaming'` por mais de 30 segundos sem novos tokens: mostrar mensagem "O assistente está demorando mais que o esperado. Tente novamente."
- [ ] Botão "Tentar novamente" no estado de timeout — limpa a mensagem em erro e permite reenviar
- [ ] Timeout não cancela o streaming se ele ainda está ativo — só exibe aviso visual
- [ ] **Verificação:** Simular lentidão → após 30s sem tokens → aviso visível; rede OK → não aparece aviso

### AC6: Testes

- [ ] `tests/unit/api/user-export.test.ts` — testar que exportação inclui todas as tabelas relevantes e respeita `user_id`
- [ ] `tests/unit/api/user-delete.test.ts` — testar que exclusão remove dados nas tabelas corretas em ordem
- [ ] Testar que usuário não consegue exportar/excluir dados de outro usuário (ownership check)
- [ ] **Verificação:** `npm test` passa; cobertura das APIs de dados inclui caso de outro `user_id`

### Exclusão em cascata — ordem importa

A ordem de exclusão deve respeitar foreign keys. Usar `createServiceClient()` para operações que escapam do RLS:

```typescript
// DELETE /api/user/account
const supabaseAdmin = await createServiceClient();

// 1. Dados financeiros
await supabaseAdmin.from('transactions').delete().eq('user_id', userId);
await supabaseAdmin.from('budgets').delete().eq('user_id', userId);
await supabaseAdmin.from('goals').delete().eq('user_id', userId);
await supabaseAdmin.from('category_dictionary').delete().eq('user_id', userId);
await supabaseAdmin.from('chat_messages').delete().eq('user_id', userId);

// 2. Assinatura Asaas (se houver)
const { data: sub } = await supabaseAdmin
  .from('subscriptions').select('asaas_subscription_id')
  .eq('user_id', userId).eq('status', 'active').maybeSingle();
if (sub?.asaas_subscription_id) {
  await cancelSubscription(sub.asaas_subscription_id); // ignora erro se não encontrar
}
await supabaseAdmin.from('subscriptions').delete().eq('user_id', userId);

// 3. Perfil e auth
await supabaseAdmin.from('profiles').delete().eq('id', userId);
await supabaseAdmin.auth.admin.deleteUser(userId);
```

### Hook offline

```typescript
// src/hooks/use-online-status.ts
'use client';
import { useState, useEffect } from 'react';

export function useOnlineStatus() {
  const [isOnline, setIsOnline] = useState(
    typeof navigator !== 'undefined' ? navigator.onLine : true
  );
  useEffect(() => {
    const handleOnline = () => setIsOnline(true);
    const handleOffline = () => setIsOnline(false);
    window.addEventListener('online', handleOnline);
    window.addEventListener('offline', handleOffline);
    return () => {
      window.removeEventListener('online', handleOnline);
      window.removeEventListener('offline', handleOffline);
    };
  }, []);
  return isOnline;
}
```

## QA Results

### Revisão Pré-Implementação — 2026-04-07
**Verdict:** CONCERNS

Um MAJOR identificado que deve ser corrigido antes de marcar a story como Done.

#### MAJOR

- **M1 — Export LGPD omite `chat_messages`**: O JSON estruturado em AC1 inclui `profile`, `transactions`, `budgets`, `goals` e `category_dictionary`, mas omite `chat_messages`. Mensagens de chat são dados pessoais (conversas financeiras com conteúdo sensível) e devem ser incluídas na exportação de portabilidade (LGPD Art. 18 III). @dev deve adicionar `chat_messages` ao payload de `/api/user/export`.

#### Informacional (sem impacto no gate)

- **Schema OK — `subscriptions` tem `ON DELETE CASCADE`**: Confirmado na migration 001 linha 152 — `references public.profiles(id) on delete cascade`. A deleção explícita no tech note de Story 4.2 é redundante mas segura.
- **`category_cache` sem `user_id` — omissão correta**: A tabela é global (cross-user) e não possui coluna `user_id`. O código de deleção de conta corretamente não tenta deletar dela. Sem ação necessária.

**Autorizado para desenvolvimento com correção de M1 durante implementação.** ✅
