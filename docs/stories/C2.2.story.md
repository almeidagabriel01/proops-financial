## Context

O Finansim já tem `recurring_rules` — mas elas são **manuais**: o usuário cria explicitamente "Netflix R$55,90 todo mês". O rastreamento de assinaturas desta story é **automático**: a IA analisa o histórico de transações e detecta padrões que parecem ser assinaturas (mesma descrição, valor similar, periodicidade mensal ou anual).

Essa distinção é crítica. O usuário não sabe que assina 12 coisas — é exatamente esse o ponto. Rocket Money ficou famoso por ajudar usuários a economizarem R$300+/mês ao descobrir assinaturas esquecidas. Copilot Money é líder global com "Subscription tracking + price change alerts". Nenhum concorrente brasileiro oferece isso.

O algoritmo de detecção olha transações do tipo `debit` dos últimos 6 meses, normaliza as descrições, e identifica grupos onde:
- A descrição normalizada é idêntica (ou ~90% similar)
- O valor é similar (±5%)
- Há pelo menos 2 ocorrências
- O intervalo entre ocorrências é ~30 dias (mensal) ou ~365 dias (anual)

Quando o valor muda entre a última e a penúltima ocorrência (reajuste detectado), o sistema gera um alerta.

**Separação clara com `recurring_rules`:**
- `recurring_rules`: criado manualmente pelo usuário, projeta pagamentos futuros em `scheduled_transactions`
- `detected_subscriptions`: detectado automaticamente, baseado em transações passadas, sem projeção futura

---

## User Story

> Como usuário,
> quero que o app detecte automaticamente meus serviços de assinatura a partir do meu histórico de transações,
> para ter uma visão consolidada de quanto gasto em assinaturas mensalmente e ser alertado quando o valor de alguma aumentar.

### AC1: Migration — tabela `detected_subscriptions`

- [ ] Nova migration `023_detected_subscriptions.sql` em `supabase/migrations/`
- [ ] Schema da tabela:
  ```sql
  CREATE TABLE detected_subscriptions (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
    description_normalized text NOT NULL,
    display_name text NOT NULL,
    current_amount numeric NOT NULL,
    previous_amount numeric,
    frequency text NOT NULL CHECK (frequency IN ('monthly', 'annual')),
    last_occurrence_date date NOT NULL,
    occurrence_count integer NOT NULL DEFAULT 2,
    price_change_detected boolean NOT NULL DEFAULT false,
    price_change_alerted_at timestamptz,
    dismissed_at timestamptz,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    UNIQUE(user_id, description_normalized)
  );
  ```
- [ ] RLS habilitado: `user_id = auth.uid()` para SELECT, UPDATE (dismiss), DELETE
- [ ] Trigger `update_updated_at()` aplicado à tabela
- [ ] **Verificação:** `supabase db push` sem erros; RLS isola registros por usuário

### AC2: Algoritmo de detecção — endpoint `POST /api/subscriptions/detect`

- [ ] Endpoint chamado automaticamente após cada importação concluída com sucesso (no final do fluxo da Edge Function ou `POST /api/import`)
- [ ] Lógica de detecção:
  1. Buscar todas as transações `debit` do usuário dos últimos 180 dias
  2. Normalizar descrições: lowercase, remover espaços extras, remover números de parcela (`\d+/\d+`), remover datas
  3. Agrupar por `description_normalized`
  4. Para cada grupo com ≥ 2 transações:
     - Calcular intervalo médio entre ocorrências
     - Mensal: intervalo médio entre 20 e 40 dias
     - Anual: intervalo médio entre 330 e 400 dias
     - Variação de valor: ≤ 5% entre ocorrências (exceto reajuste detectado)
  5. Para cada grupo que atende os critérios: upsert em `detected_subscriptions`
  6. Se `current_amount` mudou mais de 2% em relação ao `previous_amount`: setar `price_change_detected = true`
- [ ] Upsert em `detected_subscriptions` com `ON CONFLICT (user_id, description_normalized) DO UPDATE`
- [ ] **Verificação:** importar extrato com 3 cobranças mensais de "SPOTIFY BRAZIL" com valores similares → registro criado em `detected_subscriptions`

### AC3: API — `GET /api/subscriptions`

- [ ] Retorna todas as assinaturas detectadas do usuário, ordenadas por `current_amount DESC`
- [ ] Inclui campos: `id`, `display_name`, `current_amount`, `previous_amount`, `frequency`, `last_occurrence_date`, `occurrence_count`, `price_change_detected`, `dismissed_at`
- [ ] Filtra assinaturas com `dismissed_at IS NOT NULL` (descartadas) por padrão
- [ ] Query param `?include_dismissed=true` retorna todas (para histórico)
- [ ] Inclui total: `{ subscriptions: [...], total_monthly: number, total_annual_equivalent: number }`
- [ ] **Verificação:** GET retorna lista correta; total_monthly soma apenas assinaturas mensais + anuais/12

### AC4: API — `PATCH /api/subscriptions/[id]/dismiss`

- [ ] Marca uma assinatura como descartada (`dismissed_at = now()`)
- [ ] Usado quando o usuário quer remover da lista (ex: "esse pagamento não é assinatura")
- [ ] **Verificação:** PATCH → assinatura não aparece mais no GET padrão

### AC5: UI — seção "Assinaturas Detectadas" em `/financeiro/recorrentes`

- [ ] Nova aba ou seção "Assinaturas" na página `/financeiro/recorrentes`

- [ ] Header da seção mostra total mensal de assinaturas: "Total em assinaturas: R$XXX,XX/mês"
- [ ] Empty state: "Nenhuma assinatura detectada. Importe mais extratos para que o sistema identifique seus serviços recorrentes."
- [ ] Assinaturas com reajuste detectado aparecem no topo da lista (sort: price_change_detected DESC, current_amount DESC)
- [ ] **Verificação:** assinatura com `price_change_detected = true` aparece com badge vermelho e no topo; clicar "Descartar" → remove da lista

### AC6: Card de resumo de assinaturas no Dashboard

## Tech Notes for @dev

### Migration

```sql
-- supabase/migrations/023_detected_subscriptions.sql
CREATE TABLE detected_subscriptions (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  description_normalized text NOT NULL,
  display_name text NOT NULL,
  current_amount numeric(10,2) NOT NULL,
  previous_amount numeric(10,2),
  frequency text NOT NULL CHECK (frequency IN ('monthly', 'annual')),
  last_occurrence_date date NOT NULL,
  occurrence_count integer NOT NULL DEFAULT 2,
  price_change_detected boolean NOT NULL DEFAULT false,
  price_change_alerted_at timestamptz,
  dismissed_at timestamptz,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE(user_id, description_normalized)
);

ALTER TABLE detected_subscriptions ENABLE ROW LEVEL SECURITY;

CREATE POLICY "users can view and manage their subscriptions"
  ON detected_subscriptions FOR ALL
  USING (user_id = auth.uid())
  WITH CHECK (user_id = auth.uid());

CREATE TRIGGER update_detected_subscriptions_updated_at
  BEFORE UPDATE ON detected_subscriptions
  FOR EACH ROW EXECUTE FUNCTION update_updated_at();
```

### Algoritmo de detecção (TypeScript)

```typescript
// src/lib/subscriptions/detect-subscriptions.ts

function normalizeDescription(desc: string): string {
  return desc
    .toLowerCase()
    .replace(/\d+\/\d+/g, '')        // remove "1/12", "3/6"
    .replace(/\d{2}\/\d{2}\/\d{4}/g, '') // remove datas
    .replace(/\s+/g, ' ')
    .trim();
}

function isMonthlyInterval(daysBetween: number): boolean {
  return daysBetween >= 20 && daysBetween <= 40;
}

function isAnnualInterval(daysBetween: number): boolean {
  return daysBetween >= 330 && daysBetween <= 400;
}

function isAmountSimilar(a: number, b: number): boolean {
  return Math.abs(a - b) / Math.max(a, b) <= 0.05; // ±5%
}
```

### Integração pós-import

No callback de conclusão da importação (provavelmente onde `status = 'completed'` é setado), fazer `fetch('/api/subscriptions/detect', { method: 'POST' })` com o `userId`. Pode ser fire-and-forget (não bloquear a resposta ao usuário).

### Exibição do `display_name`

Para gerar o `display_name` a partir da `description_normalized`, capitalizar a primeira letra de cada palavra e remover prefixos de banco comuns ("COMPRA DEBITO", "PAGAMENTO", "TED", etc.). Manter como best-effort — o usuário não pode editar no MVP.

## Out of Scope

- Edição do nome exibido da assinatura pelo usuário
- Cancelamento automático de assinaturas (como Rocket Money)
- Link para cancelar a assinatura (ex: link para site do serviço)
- Projeção de gastos futuros com assinaturas (isso é o fluxo de caixa)
- Assinaturas de cartão de crédito (só detecta em transações de débito/conta corrente no MVP)
- Notificação push de reajuste detectado (pode ser adicionado como extensão desta story)
