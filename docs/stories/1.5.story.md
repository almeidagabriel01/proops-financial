## Context

O usuario so consegue adicionar transacoes importando extratos OFX/CSV. Isso deixa de fora casos cruciais: salario depositado em conta diferente, pagamentos em dinheiro, transferencias entre contas proprias, despesas de cartao de credito nao importado, reembolsos.

Sem essa story, o dashboard nao reflete a realidade financeira do usuario — apenas o que o banco exporta. Isso limita o valor central do produto: "saber para onde vai seu dinheiro".

**Disponibilidade:** Basic e Pro (UI form). A funcao `create_transaction` via chat IA permanece exclusiva do Pro via function calling (FR28, Story 3.x) — sao features distintas e complementares.

**Decisao de produto:** Transacoes manuais usam `bank_account_id` de uma conta existente do usuario ou de uma conta virtual "Manual" criada automaticamente no primeiro uso. Isso preserva a integridade referencial sem forcar o usuario a criar uma conta bancaria antes de adicionar uma despesa avulsa.

**Bloqueio identificado:** Story 2.2 (aprendizado de categoria) e Story 3.x (function calling) dependem de edicao e exclusao de transacoes. Esta story desbloqueia ambas.

**PRD refs:** FR5 (historico ilimitado em ambos os planos), FR28 (referencia — function calling Pro e separado)
**Arch refs:** `transactions` table schema (section 3.1), RLS policies (section 3.2), Component Strategy (section 7.1)

## User Story

> Como usuario,
> quero adicionar, editar e excluir transacoes manualmente,
> para que meu dashboard reflita toda a minha realidade financeira, incluindo gastos em dinheiro e receitas que nao aparecem no extrato bancario.

## Acceptance Criteria

### AC1: Adicionar transacao manual
- [ ] Botao "+" ou "Adicionar" visivel na pagina `/transactions` (e no dashboard como CTA secundario)
- [ ] Formulario com campos obrigatorios:
  - **Data** — date picker, default hoje
  - **Descricao** — texto livre, max 255 chars
  - **Valor** — numerico, positivo, formato BRL (virgula como decimal)
  - **Tipo** — toggle ou select: Receita / Despesa
  - **Categoria** — select com as 14 categorias fixas, default 'outros'
- [ ] Campo opcional: **Conta** — select com contas bancarias existentes do usuario; se nenhuma existe, criar automaticamente conta "Manual" (banco_name = 'Manual')
- [ ] Ao salvar: inserir na tabela `transactions` com `external_id = 'manual_' + crypto.randomUUID()`, `category_source = 'user'`, `import_id = null`
- [ ] Transacao aparece imediatamente na lista e no dashboard (invalidar cache / reload)
- [ ] **Verificacao:** Adicionar despesa "Aluguel" R$1.200,00; aparece na lista e soma corretamente no dashboard

### AC2: Formulario com validacao
- [ ] Data: nao pode ser mais de 10 anos no passado nem no futuro (aviso, nao bloqueio)
- [ ] Valor: > 0, maximo R$999.999,99; rejeitar zero ou negativo com mensagem clara
- [ ] Descricao: obrigatoria, nao pode ser apenas espacos em branco
- [ ] Categoria: obrigatoria, deve ser uma das 14 categorias validas
- [ ] Erros exibidos inline abaixo de cada campo (nao toast)
- [ ] Botao "Salvar" desabilitado enquanto formulario tem erros ou esta enviando
- [ ] **Verificacao:** Tentar salvar com valor 0 → erro inline; tentar sem descricao → erro inline

### AC3: Editar transacao existente
- [ ] Na lista de transacoes, cada item tem menu de contexto (3 pontos "..." ou swipe) com opcoes: "Editar" e "Excluir"
- [ ] "Editar" abre o mesmo formulario pre-preenchido com dados atuais
- [ ] Campos editaveis: data, descricao, valor, tipo, categoria (conta nao editavel)
- [ ] Ao salvar: atualizar registro na tabela `transactions`; `updated_at` atualizado automaticamente via trigger
- [ ] Tanto transacoes manuais quanto importadas podem ser editadas (campo de descricao e categoria)
- [ ] **Verificacao:** Editar categoria de "IFOOD*PEDIDO" de 'outros' para 'delivery'; mudanca persiste apos reload

### AC4: Excluir transacao
- [ ] "Excluir" exibe dialogo de confirmacao antes de deletar (nao excluir direto)
- [ ] Texto da confirmacao: "Tem certeza que deseja excluir esta transacao? Esta acao nao pode ser desfeita."
- [ ] Ao confirmar: `DELETE` na tabela `transactions` com `id` e `user_id` (RLS garante isolamento)
- [ ] Transacao removida da lista imediatamente (optimistic update ou reload)
- [ ] Dashboard atualizado apos exclusao
- [ ] **Verificacao:** Excluir transacao; confirmar; transacao desaparece da lista e totais do dashboard atualizam

### AC5: Conta "Manual" criada automaticamente
- [ ] Se usuario nao tem nenhuma `bank_account`, ao salvar a primeira transacao manual, criar automaticamente: `{ bank_name: 'Manual', account_label: 'Lancamentos manuais' }`
- [ ] Conta "Manual" aparece no select de conta do formulario como opcao default quando nenhuma outra conta existe
- [ ] Se usuario ja tem contas importadas, o select mostra todas as contas + "Manual" (criada se necessario)
- [ ] **Verificacao:** Usuario sem import; adicionar transacao manual; conta "Manual" e criada; transacao associada a ela

### AC6: Integracao com dashboard e lista
- [ ] Transacoes manuais aparecem na lista `/transactions` identicamente as importadas
- [ ] Badge ou indicador visual sutil distingue transacoes manuais das importadas (ex: icone de lapis ou tag "Manual")
- [ ] Totais no dashboard incluem transacoes manuais
- [ ] Filtros de periodo, tipo e busca funcionam sobre transacoes manuais
- [ ] **Verificacao:** Adicionar receita "Salario" R$5.000,00; dashboard mostra valor atualizado no total de receitas

### AC7: API Route server-side para mutations
- [ ] `POST /api/transactions` — criar transacao manual (validacao server-side)
- [ ] `PATCH /api/transactions/[id]` — editar transacao (valida que `user_id` = usuario autenticado via RLS)
- [ ] `DELETE /api/transactions/[id]` — excluir transacao (valida posse via RLS)
- [ ] Todas as rotas: autenticacao obrigatoria; retornar 401 se nao autenticado
- [ ] Validacao de schema server-side (nao confiar apenas no frontend)
- [ ] `category` deve ser uma das 14 categorias validas (validar no servidor)
- [ ] **Verificacao:** Chamar DELETE sem autenticacao → 401; chamar com id de outro usuario → 404 (RLS oculta)

### AC8: Feedback visual e UX mobile
- [ ] Formulario abre como sheet/modal bottom-sheet em mobile (nao nova pagina)
- [ ] Loading state no botao "Salvar" durante envio (spinner + texto "Salvando...")
- [ ] Toast de sucesso apos criar/editar/excluir ("Transacao adicionada", "Alteracoes salvas", "Transacao excluida")
- [ ] Formulario fecha automaticamente apos sucesso
- [ ] **Verificacao:** Em viewport 360px, formulario e usavel sem scroll horizontal; bottom sheet fecha apos salvar

## Tech Notes para @dev

### external_id para transacoes manuais
```typescript
// Formato padrao — distingue de OFX (FITID) e CSV (SHA-256)
const externalId = `manual_${crypto.randomUUID()}`;
```
Nao ha risco de colisao com transacoes importadas pois FITID nunca comeca com "manual_" e SHA-256 e hexadecimal puro.

### API Routes necessarias
```
POST   /api/transactions          → criar
PATCH  /api/transactions/[id]     → editar
DELETE /api/transactions/[id]     → excluir
```

### Schema insert (sem mudancas no banco)
```typescript
// POST /api/transactions body
{
  date: string,           // ISO 8601 YYYY-MM-DD
  description: string,    // max 255
  amount: number,         // positivo (sinal definido por 'type')
  type: 'credit' | 'debit',
  category: Category,     // uma das 14 fixas
  bank_account_id?: string // opcional; criar 'Manual' se ausente
}

// Insert na tabela transactions
{
  user_id: auth.uid(),
  bank_account_id: resolvedAccountId,
  import_id: null,          // transacao manual nao tem import
  external_id: `manual_${crypto.randomUUID()}`,
  date,
  description,
  amount: type === 'debit' ? -Math.abs(amount) : Math.abs(amount),
  type,
  category,
  category_source: 'user',  // usuario escolheu a categoria
  category_confidence: null,
}
```

### Conta "Manual" — logica de upsert
```typescript
// Criar ou buscar conta Manual existente
async function resolveManualAccount(supabase, userId: string): Promise<string> {
  const { data: existing } = await supabase
    .from('bank_accounts')
    .select('id')
    .eq('user_id', userId)
    .eq('bank_name', 'Manual')
    .maybeSingle();

  if (existing) return existing.id;

  const { data: created } = await supabase
    .from('bank_accounts')
    .insert({ user_id: userId, bank_name: 'Manual', account_label: 'Lancamentos manuais' })
    .select('id')
    .single();

  return created!.id;
}
```

### RLS — sem mudancas necessarias
A policy existente "Users can CRUD own transactions" cobre insert, update e delete via `user_id = auth.uid()`. Nao e necessaria nova migration.

### Formulario — componente reutilizavel
```
src/components/transactions/transaction-form.tsx
```
Usado tanto para criar (campos em branco) quanto para editar (campos pre-preenchidos via prop `initialValues`). Aceita `onSuccess` callback.

### shadcn/ui — componentes necessarios
- `Sheet` / `Drawer` — bottom sheet em mobile para o formulario
- `Dialog` — confirmacao de exclusao
- `Select` — categoria e conta
- `DatePicker` (ou `Input type="date"`) — campo de data
- `Toast` / `Sonner` — feedback de sucesso/erro

### Valor BRL no input
Usar input `type="text"` com mascara (ou `type="number"` com step="0.01"). Converter virgula para ponto antes de enviar para API. Exibir erro se nao for numero valido.

### Indicador visual de transacao manual
```tsx
// Em transaction-item.tsx — adicionar badge condicional
{transaction.import_id === null && (
  <span className="text-xs text-muted-foreground">Manual</span>
)}
```

### Invalidar cache apos mutation
Como a lista em `/transactions` usa hook `useTransactions` (cliente), chamar `refetch()` ou `router.refresh()` apos sucesso. Dashboard e Server Component — usar `router.refresh()` para re-render server-side.
