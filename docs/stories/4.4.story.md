## Context

O produto é web mobile-first. 85% dos usuários brasileiros estão em Android — muitos em dispositivos de entrada com hardware limitado e conexões instáveis. Dois problemas urgentes antes do lançamento:

**PWA incompleto:** O `manifest.json` existe mas sem service worker o app não pode ser instalado na home screen (Add to Home Screen) nem funciona offline. Nubank e PicPay têm essa experiência — usuários esperam isso de um app financeiro.

**Performance desconhecida:** Não rodamos Lighthouse nem medimos Core Web Vitals em nenhum momento do desenvolvimento. Podemos estar enviando bundles desnecessariamente grandes para dispositivos de entrada.

**Esta story não é otimização prematura** — é garantia mínima de qualidade antes de colocar usuários reais no produto.

---

## User Story

> Como usuário em smartphone Android de entrada,
> quero poder instalar o app na minha home screen e ter uma experiência rápida mesmo com conexão limitada,
> para que o app pareça nativo e carregue rápido como os apps de banco que já uso.

## Acceptance Criteria

### AC1: Service Worker — instalação PWA

- [ ] Service worker registrado via `next-pwa` ou implementação manual com Workbox
- [ ] Estratégia de cache para assets estáticos: `CacheFirst` (JS, CSS, imagens)
- [ ] Estratégia de cache para API routes: `NetworkFirst` (dados sempre frescos quando online)
- [ ] Página `/offline` — exibida pelo service worker quando offline e sem cache disponível
- [ ] `manifest.json` atualizado (ou criado se não existir):
  - `name`: "Finansim" (ou nome real do app)
  - `short_name`: "Finansim"
  - `start_url`: "/dashboard"
  - `display`: "standalone"
  - `theme_color`: cor primária do design system
  - `background_color`: branco ou cor de splash
  - `icons`: array com pelo menos 192x192 e 512x512 (PNG, pode ser ícone placeholder no MVP)
- [ ] Prompt de instalação não-intrusivo: banner no dashboard após 2+ visitas ("Adicionar à tela inicial")
- [ ] **Verificação:** Chrome Android → três pontos → "Adicionar à tela inicial" aparece; ícone instalado na home screen

### AC2: Core Web Vitals — baseline

- [ ] Rodar Lighthouse CI no ambiente de staging/preview antes de marcar done
- [ ] Targets mínimos (mobile):
  - LCP (Largest Contentful Paint): < 2.5s
  - FID/INP (Interaction to Next Paint): < 200ms
  - CLS (Cumulative Layout Shift): < 0.1
  - Performance Score: >= 75
- [ ] Documentar baseline em `docs/performance-baseline.md` com screenshot do relatório Lighthouse
- [ ] Identificar e corrigir os 2-3 maiores ofensores se score < 75
- [ ] **Verificação:** Lighthouse mobile no deploy de preview retorna score >= 75

### AC3: Bundle size — auditoria e otimização

- [ ] Rodar `npm run build` e analisar o output de tamanhos de bundle
- [ ] Verificar se alguma rota tem First Load JS > 300KB — se sim, investigar e otimizar
- [ ] `next/dynamic` para componentes pesados que não são críticos no primeiro paint:
  - Gráficos do dashboard (recharts/chart.js) — carregar apenas quando visíveis
  - Componente de importação (dropzone) — lazy se não for a página inicial
- [ ] Verificar se alguma biblioteca grande foi importada desnecessariamente (lodash completo, etc.)
- [ ] **Verificação:** `npm run build` → nenhuma rota com First Load JS > 300KB; se havia, mostrar before/after

### AC4: Imagens e fontes otimizadas

- [ ] Todas as imagens usando `next/image` com `width`, `height` e `priority` definidos
- [ ] Fontes do Google (se usadas) carregadas via `next/font` com `display: 'swap'`
- [ ] Nenhuma imagem acima de 100KB sem otimização
- [ ] Verificar se há layout shifts causados por fontes ou imagens sem dimensões definidas (CLS)
- [ ] **Verificação:** Lighthouse não reporta "Properly size images" ou "Eliminate render-blocking resources" como crítico

### AC5: Página de loading e skeleton screens

- [ ] Todos os estados de loading nas páginas principais usam Skeleton (já implementados nos Epics anteriores) — verificar que não há spinners genéricos ou telas em branco
- [ ] `loading.tsx` no App Router para as rotas principais (`/dashboard`, `/transactions`, `/chat`)
- [ ] Splash screen via CSS durante o carregamento inicial (enquanto Next.js hidrata)
- [ ] **Verificação:** Recarregar qualquer página → skeleton visível durante carregamento, sem flash de conteúdo em branco

### AC6: Testes de performance automatizados

- [ ] Script `npm run lighthouse` que roda Lighthouse CI contra o servidor de dev local
- [ ] Configuração em `lighthouserc.js` com os thresholds do AC2
- [ ] Script falha (exit code 1) se qualquer threshold for violado
- [ ] Documentação no README de como rodar o check de performance manualmente
- [ ] **Verificação:** `npm run lighthouse` passa com thresholds configurados

## Tech Notes for @dev

### @serwist/next (substituto oficial do next-pwa para App Router)

`next-pwa` é incompatível com Next.js 13+ App Router e está abandonado. Usar `@serwist/next`, fork ativo com suporte nativo ao App Router.

```bash
npm install @serwist/next serwist
```

Criar `src/sw.ts` (service worker source):

```typescript
// src/sw.ts
import { defaultCache } from '@serwist/next/worker';
import { installSerwist } from 'serwist';

installSerwist({
  precacheEntries: self.__SW_MANIFEST,
  skipWaiting: true,
  clientsClaim: true,
  navigationPreload: false,
  runtimeCaching: defaultCache,
});
```

```typescript
// next.config.ts
import withSerwist from '@serwist/next';

const withPWA = withSerwist({
  swSrc: 'src/sw.ts',
  swDest: 'public/sw.js',
  disable: process.env.NODE_ENV === 'development',
});

export default withPWA({
  // config existente do Next.js
});
```

Adicionar ao `.gitignore`: `public/sw.js`, `public/sw.js.map`, `public/workbox-*.js`.

### Prompt de instalação

```typescript
// src/hooks/use-pwa-install.ts
'use client';
import { useState, useEffect } from 'react';

export function usePWAInstall() {
  const [deferredPrompt, setDeferredPrompt] = useState<BeforeInstallPromptEvent | null>(null);
  
  useEffect(() => {
    const handler = (e: Event) => {
      e.preventDefault();
      setDeferredPrompt(e as BeforeInstallPromptEvent);
    };
    window.addEventListener('beforeinstallprompt', handler);
    return () => window.removeEventListener('beforeinstallprompt', handler);
  }, []);
  
  const install = async () => {
    if (!deferredPrompt) return;
    deferredPrompt.prompt();
    await deferredPrompt.userChoice;
    setDeferredPrompt(null);
  };
  
  return { canInstall: !!deferredPrompt, install };
}
```

## QA Results

### Revisão Pré-Implementação — 2026-04-07
**Verdict:** CONCERNS

Um MAJOR que impede o uso do pacote recomendado no tech note.

#### MAJOR

- **M1 — `next-pwa` incompatível com Next.js 16.x App Router**: O projeto usa Next.js `16.2.2` (confirmado em `package.json`). O pacote `next-pwa` (npm) é unmaintained (último commit em 2022) e **não suporta App Router**. A configuração via `withPWA()` no `next.config.ts` como descrita no tech note causará falha de build ou comportamento incorreto do service worker.

  **Substituto obrigatório:** `@serwist/next` — fork ativo do `next-pwa` com suporte nativo ao App Router. API é similar:
  ```bash
  npm install @serwist/next serwist
  ```
  ```typescript
  // next.config.ts
  import withSerwist from '@serwist/next';
  const withPWA = withSerwist({ swSrc: 'src/sw.ts', swDest: 'public/sw.js', disable: process.env.NODE_ENV === 'development' });
  export default withPWA({ /* next config */ });
  ```
  @dev deve usar `@serwist/next` em vez de `next-pwa` para todos os itens de AC1.

#### Observações (sem impacto no gate)

- Lighthouse targets (score >= 75, LCP < 2.5s, CLS < 0.1) são realistas para MVP mobile-first com Vercel Edge.
- Dynamic imports para gráficos (AC3) é correto — Recharts é ~400KB e não deve estar no LCP path.

**Autorizado para desenvolvimento com substituição de `next-pwa` por `@serwist/next`.** ✅
