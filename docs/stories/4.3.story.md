## Context

O produto vai para produção sem nenhum mecanismo de observabilidade além de `console.log`. Quando algo quebrar em produção (e vai quebrar), não teremos visibilidade do erro, do contexto, nem de quantos usuários foram afetados.

**O que está faltando:**
- Erros JavaScript no frontend desaparecem silenciosamente
- Erros nas API routes aparecem apenas nos logs da Vercel (sem agrupamento, sem alertas)
- O endpoint `/api/health` verifica Supabase mas não verifica Claude API nem o estado real das dependências
- Não há nenhum dado sobre uso real: quantos usuários ativos, quantas importações por dia, taxa de conversão do trial

**Esta story instala os instrumentos mínimos** para operar com confiança em produção — sem precisar de infraestrutura complexa ou dashboards elaborados.

---

## User Story

> Como desenvolvedor,
> quero saber quando algo quebra em produção antes que o usuário me reporta,
> e quero métricas básicas de uso para tomar decisões de produto,
> para que eu possa operar e iterar com confiança após o lançamento.

## Acceptance Criteria

### AC1: Sentry — frontend

- [ ] `@sentry/nextjs` instalado e configurado
- [ ] `sentry.client.config.ts` e `sentry.server.config.ts` na raiz
- [ ] `SENTRY_DSN`, `SENTRY_ORG` e `SENTRY_PROJECT` em `.env.local` (valores reais) e `.env.example` (template sem valores)
- [ ] Erros JavaScript não capturados → enviados ao Sentry automaticamente
- [ ] Identificação de usuário: `Sentry.setUser({ id: user.id })` após login; `Sentry.setUser(null)` após logout
- [ ] Em desenvolvimento: Sentry desabilitado (`enabled: process.env.NODE_ENV === 'production'`)
- [ ] Source maps enviados no build (next.config.ts com Sentry plugin) — stack traces legíveis
- [ ] **Verificação:** Em produção, `throw new Error('test')` em qualquer componente → aparece no Sentry dashboard

### AC2: Sentry — API routes

- [ ] Wrapper `withSentry` ou `Sentry.captureException` nos catch blocks das API routes críticas:
  - `/api/chat` — erros de streaming
  - `/api/import` — erros de parsing e categorização
  - `/api/webhook/asaas` — erros de processamento de pagamento
  - `/api/audio` — erros de transcrição
- [ ] Context adicional em cada captura: `user_id`, `plan`, nome da operação
- [ ] Erros de negócio esperados (401, 429, 403) NÃO devem ir ao Sentry — apenas erros 5xx inesperados
- [ ] **Verificação:** Erro simulado em `/api/chat` com `ANTHROPIC_API_KEY` inválida → aparece no Sentry com user context

### AC3: Health check melhorado (`/api/health`)

- [ ] Health check atual: só verifica conexão com Supabase
- [ ] Melhorado para verificar:
  - Supabase: query simples `SELECT 1` — `{ status: 'ok' | 'error', latencyMs: number }`
  - Claude API: verificar se `ANTHROPIC_API_KEY` está configurada — `{ status: 'configured' | 'missing' }`
  - OpenAI API (Whisper): verificar se `OPENAI_API_KEY` está configurada — `{ status: 'configured' | 'missing' }`
  - Asaas: verificar se `ASAAS_API_KEY` está configurada — `{ status: 'configured' | 'missing' }`
- [ ] Response estruturada:
  ```json
  {
    "status": "ok" | "degraded" | "error",
    "version": "1.0.0",
    "timestamp": "2026-04-07T20:00:00.000Z",
    "dependencies": {
      "supabase": { "status": "ok", "latencyMs": 45 },
      "anthropic": { "status": "configured" },
      "openai": { "status": "missing" },
      "asaas": { "status": "configured" }
    }
  }
  ```
- [ ] `status: "degraded"` se alguma API key está faltando mas Supabase está OK
- [ ] `status: "error"` se Supabase não responde
- [ ] **Verificação:** `GET /api/health` → JSON com status de cada dependência

### AC4: Analytics básico — eventos de uso

- [ ] `src/lib/analytics/events.ts` — funções para registrar eventos na tabela `analytics_events`
- [ ] Migration criando tabela `analytics_events (id, user_id, event_name, properties jsonb, created_at)`
- [ ] RLS: usuários não podem ler analytics_events (write-only via service role)
- [ ] Eventos a capturar:
  - `import_completed` — `{ file_format, transaction_count, duration_ms }`
  - `chat_message_sent` — `{ plan, model, query_count_after }`
  - `trial_started` — `{ plan: 'pro' }`
  - `subscription_created` — `{ plan, billing_cycle }`

- [ ] Captura server-side nas API routes relevantes (nunca no frontend para evitar ad-blockers)
- [ ] **Verificação:** Após importar extrato → `analytics_events` tem registro `import_completed`

### AC5: Query de métricas no Supabase (sem dashboard externo)

- [ ] Acessível apenas com service role (sem RLS para essa view)
- [ ] Documentação em `docs/analytics.md` de como consultar manualmente via Supabase Studio
- [ ] **Verificação:** Query na view retorna dados corretos após eventos de teste inseridos

## Tech Notes for @dev

### Instalação do Sentry

```bash
npx @sentry/wizard@latest -i nextjs
```

O wizard configura automaticamente `next.config.ts`, `sentry.client.config.ts` e `sentry.server.config.ts`. Aceitar todas as opções padrão.

### Captura de erro nas API routes

```typescript
// Padrão para API routes críticas
import * as Sentry from '@sentry/nextjs';

try {
  // ... lógica
} catch (err) {
  // Só captura se for erro inesperado (não HTTP 4xx controlados)
  Sentry.captureException(err, {
    extra: { userId: user.id, operation: 'chat_stream' },
  });
  return NextResponse.json({ error: 'Erro interno' }, { status: 500 });
}
```

### Analytics — inserção server-side

```typescript
// src/lib/analytics/events.ts
export async function trackEvent(
  supabaseAdmin: SupabaseClient,
  userId: string,
  eventName: string,
  properties: Record<string, unknown> = {},
) {
  // fire-and-forget — nunca bloquear a resposta principal
  supabaseAdmin
    .from('analytics_events')
    .insert({ user_id: userId, event_name: eventName, properties })
    .then(() => {}) // ignorar erro — analytics não deve quebrar o fluxo
    .catch(() => {});
}
```

### Migration analytics

```sql
-- supabase/migrations/009_analytics.sql
CREATE TABLE IF NOT EXISTS public.analytics_events (
  id          uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id     uuid REFERENCES public.profiles(id) ON DELETE CASCADE,
  event_name  text NOT NULL,
  properties  jsonb DEFAULT '{}',
  created_at  timestamptz DEFAULT now()
);

-- Usuários não leem seus eventos (write-only via service role)
ALTER TABLE public.analytics_events ENABLE ROW LEVEL SECURITY;
CREATE POLICY "analytics_no_read" ON public.analytics_events
  FOR SELECT USING (false);

-- Index para queries de aggregate
CREATE INDEX analytics_events_created_at_idx ON public.analytics_events(created_at);
CREATE INDEX analytics_events_event_name_idx ON public.analytics_events(event_name);
```
