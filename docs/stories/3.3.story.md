## Context

Com o chat básico funcionando (Story 3.2), esta story adiciona as features exclusivas do plano Pro que tornam o assistente ativo, não apenas consultivo:

**Function Calling (Pro):** O Sonnet pode executar ações reais na plataforma — criar transação, recategorizar, criar orçamento, criar objetivo financeiro, excluir transação (com confirmação). Usa Vercel AI SDK `streamText()` com `tools`. O LLM decide quando chamar uma ferramenta baseado na mensagem do usuário.

**Entrada por Áudio (Pro):** O usuário grava uma mensagem de voz no browser (MediaRecorder), envia para `/api/audio` que transcreve via Whisper API, e o texto transcrito entra no fluxo normal de chat.

**Modo stub:** Quando OPENAI_API_KEY não configurada (para Whisper), o endpoint retorna uma transcrição simulada.

## Tech Notes for @dev

### makeProTools — factory com closure (padrão correto)

No Vercel AI SDK, `tool.execute` recebe apenas `(args, { abortSignal })` — sem userId ou supabase em options. O padrão correto é uma **factory function** que captura contexto via closure:

```typescript
// src/lib/ai/tools/index.ts
import { tool } from 'ai'
import { z } from 'zod'
import type { SupabaseClient } from '@supabase/supabase-js'
import { CATEGORIES } from '@/lib/billing/plans'
import { formatCurrency } from '@/lib/utils/format'

export function makeProTools(userId: string, supabaseAdmin: SupabaseClient) {
  return {
    create_transaction: tool({
      description: 'Cria uma receita ou despesa manualmente para o usuário',
      parameters: z.object({
        date: z.string().describe('Data no formato YYYY-MM-DD'),
        description: z.string().describe('Descrição da transação'),
        amount: z.number().describe('Valor positivo para receita, negativo para despesa'),
        category: z.enum(CATEGORIES),
      }),
      execute: async ({ date, description, amount, category }) => {
        // userId e supabaseAdmin capturados do closure — sem parâmetros extras
        const { data, error } = await supabaseAdmin
          .from('transactions')
          .insert({
            user_id: userId,   // closure: seguro, não vem do LLM
            date, description, amount,
            type: amount >= 0 ? 'credit' : 'debit',
            category,
            category_source: 'user',
          })
          .select('id')
          .single()
        if (error) return { success: false, error: error.message }
        return {
          success: true,
          transactionId: data.id,
          message: `Transação criada: ${description} — ${formatCurrency(Math.abs(amount))}`,
        }
      },
    }),
    // ... demais tools seguem o mesmo padrão
  }
}
```

### Uso no /api/chat/route.ts

```typescript
import { createServiceRoleClient } from '@/lib/supabase/service'  // service role
import { makeProTools } from '@/lib/ai/tools'

// Dentro do handler POST:
const supabaseAdmin = createServiceRoleClient()
const tools = tier === 'pro' ? makeProTools(user.id, supabaseAdmin) : undefined

const result = await streamText({
  model: anthropic(model),
  system: buildSystemPrompt(context),
  messages,
  tools,            // undefined para Basic = LLM nunca recebe tool definitions
  maxTokens: 1024,
})
```

> **Por que service role nas tools?** As tool execute() fazem writes em nome do usuário. O service role client com `user_id` explícito no WHERE garante autorização sem depender de RLS — e permite que o webhook handler reutilize o mesmo padrão. O client de usuário com RLS é suficiente para reads (contexto financeiro), mas service role é necessário para writes das tools.

### Audio route — multipart
```typescript
// src/app/api/audio/route.ts
import FormData from 'form-data'
import fetch from 'node-fetch' // ou usar global fetch do Node 18+

export async function POST(req: Request) {
  // Auth + plan check
  const formData = await req.formData()
  const file = formData.get('file') as File

  if (!process.env.OPENAI_API_KEY) {
    return NextResponse.json({ transcript: '[Transcrição de demonstração] Quanto gastei esse mês?' })
  }

  const whisperForm = new FormData()
  whisperForm.append('file', file, { filename: 'audio.webm', contentType: file.type })
  whisperForm.append('model', 'whisper-1')
  whisperForm.append('language', 'pt')

  const res = await fetch('https://api.openai.com/v1/audio/transcriptions', {
    method: 'POST',
    headers: { Authorization: `Bearer ${process.env.OPENAI_API_KEY}`, ...whisperForm.getHeaders() },
    body: whisperForm,
  })

  const { text } = await res.json()
  return NextResponse.json({ transcript: text })
}
```

### Issues

**~~B1 — PRO_TOOLS como static export é incompatível com Vercel AI SDK~~ — RESOLVIDO**
- **Resolution:** AC1 reescrito com `makeProTools(userId, supabaseAdmin)` factory function usando closure. `supabaseAdmin` é service role client criado dentro da API route e passado para a factory. Padrão correto implementado na tech note de `/api/chat/route.ts`.

**~~m1 — Cliente Supabase nas tools: user client vs service role não especificado~~ — RESOLVIDO**
- **Resolution:** AC1 agora especifica explicitamente que `supabaseAdmin` dentro de `makeProTools` é o service role client (`createServiceRoleClient()`). Tech note em AC1 explica o motivo (writes de tools bypassam RLS via userId explícito).

**m2 — Audio route tech note usa imports desnecessários**
- **Severity:** MINOR
- **Category:** code
- **Description:** Tech note importa `FormData from 'form-data'` e `fetch from 'node-fetch'` — ambos desnecessários em Next.js 14+ (Node 18+) que tem native fetch. Para Whisper, o pacote `openai` (listado como dependência) trata o upload nativamente: `openai.audio.transcriptions.create({ file, model: 'whisper-1', language: 'pt' })`.
- **Recommendation:** Usar o SDK OpenAI: `const openai = new OpenAI(); const { text } = await openai.audio.transcriptions.create({ file: audioFile, model: 'whisper-1', language: 'pt' })`.
