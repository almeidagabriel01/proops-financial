## Context

Com o chat e o sistema de planos funcionando, esta story fecha o ciclo de monetização: um usuário pode assinar o plano Basic ou Pro via Asaas (cartão, boleto, Pix), e o acesso às features é ativado/desativado automaticamente conforme o status da assinatura.

**Schema já existe:** tabela `subscriptions`, trigger `sync_plan_from_subscription`, coluna `profiles.audio_enabled`. A migration 003 já aplicou toda a estrutura de dados.

**O que esta story constrói:**
1. `src/lib/billing/asaas.ts` — cliente HTTP para a API Asaas
2. `src/app/api/webhook/asaas/route.ts` — endpoint de webhook com validação HMAC
3. `src/lib/billing/webhook-handler.ts` — processamento de eventos Asaas
4. Tela de checkout e gerenciamento de assinatura em `/settings?tab=plano`

---

## User Story

> Como usuário,
> quero assinar o plano Basic ou Pro com cartão, boleto ou Pix
> e gerenciar minha assinatura diretamente no app,
> para que eu tenha controle total sobre meu plano sem precisar de suporte.

- [ ] Validar token de acesso: header `asaas-access-token` deve ser igual ao valor de `ASAAS_WEBHOOK_TOKEN` no `.env` — **comparação de string estática, não HMAC criptográfico** (mecanismo do Asaas v3)

- [ ] **Verificação:** Evento `PAYMENT_CONFIRMED` com `asaas_subscription_id` válido → `profiles.plan = 'pro'`

## Tech Notes for @dev

### Asaas client
```typescript
// src/lib/billing/asaas.ts
const ASAAS_BASE = process.env.ASAAS_BASE_URL ?? 'https://api.asaas.com/v3'

async function asaasFetch(path: string, options?: RequestInit) {
  const res = await fetch(`${ASAAS_BASE}${path}`, {
    ...options,
    headers: {
      'access_token': process.env.ASAAS_API_KEY!,
      'Content-Type': 'application/json',
      ...options?.headers,
    },
  })
  if (!res.ok) throw new Error(`Asaas error ${res.status}: ${await res.text()}`)
  return res.json()
}

export async function createCustomer(data: { name: string; email: string; cpfCnpj: string }) {
  return asaasFetch('/customers', { method: 'POST', body: JSON.stringify(data) })
}
```

### Validação de token do webhook (Asaas usa token estático)
```typescript
// src/app/api/webhook/asaas/route.ts
export async function POST(req: Request) {
  // Asaas v3 envia o token configurado no dashboard via header 'asaas-access-token'
  // Não é HMAC — é comparação direta de string
  const token = req.headers.get('asaas-access-token')
  if (!token || token !== process.env.ASAAS_WEBHOOK_TOKEN) {
    return new Response('Unauthorized', { status: 401 })
  }
  const event = await req.json()
  const supabase = createServiceRoleClient() // service role para webhook
  await handleAsaasWebhook(event, supabase)
  return new Response('OK', { status: 200 })
}
```

### Issues

**M1 — AC4 não especifica salvar asaas_subscription_id no DB durante checkout**
- **Severity:** MAJOR
- **Category:** requirements
- **Description:** O webhook handler em AC3 faz upsert em `subscriptions` usando `asaas_subscription_id` como chave de idempotência. Mas o upsert inclui `user_id: userId` — de onde vem esse userId na hora do webhook? O evento Asaas inclui `subscription.id` e `subscription.customer`. A tabela `subscriptions` não tem `asaas_customer_id`. Se o `/api/checkout` não salvar `{ user_id, asaas_subscription_id }` no banco antes de redirecionar, o webhook não consegue mapear `subscription.id` → `user_id` e falhará silenciosamente (não encontra registro para atualizar).
- **Required fix:** Adicionar ao AC4: após `createSubscription()` retornar o ID da assinatura do Asaas, salvar imediatamente um registro em `subscriptions` com `{ user_id, asaas_subscription_id, status: 'pending' }`. O webhook então localiza o registro pelo `asaas_subscription_id` e atualiza `user_id` que já está lá. Adicionar também ao Tech Notes: "O mapeamento user_id → asaas_subscription_id é estabelecido durante o checkout, não no webhook."

**M2 — "HMAC" no AC2 é terminologia incorreta para Asaas**
- **Severity:** MAJOR
- **Category:** requirements
- **Description:** AC2 diz "Validar assinatura HMAC: header `asaas-access-token`...". O Asaas não usa HMAC criptográfico — usa comparação de token estático configurado no dashboard. HMAC implica `HMAC-SHA256(payload, secret)`, que não é o que o Asaas faz. Um @dev diligente que pesquisar "webhook HMAC verification" implementará verificação criptográfica desnecessária (ou incorreta). O código de exemplo na tech note está correto (`token !== ASAAS_WEBHOOK_SECRET`), mas o AC contradiz a implementação.
- **Correction:** Alterar AC2 para: "Validar token de acesso: header `asaas-access-token` deve ser igual ao valor configurado em `ASAAS_WEBHOOK_SECRET`." Remover a palavra "HMAC" do AC.

**m3 — Out of scope contradiz AC4 em planos anuais**
- **Severity:** MINOR
- **Category:** requirements
- **Description:** Out of Scope diz "Planos anuais no checkout MVP (implementar mensal primeiro)". AC4 payload inclui `'basic_monthly' | 'basic_yearly' | 'pro_monthly' | 'pro_yearly'`. O @dev implementará os 4 ou apenas os 2 mensais?
- **Recommendation:** Alinhar AC4 com Out of Scope: `{ planKey: 'basic_monthly' | 'pro_monthly' }` para MVP. Anuais são tech debt explícito.

**m4 — AC7 não especifica mudança necessária em getEffectiveTier para período de graça**
- **Severity:** MINOR
- **Category:** requirements
- **Description:** AC7 diz "Verificação em `/api/chat` e `/api/audio`: `getEffectiveTier()` considera `past_due` como Pro por 3 dias". Mas `getEffectiveTier(plan, trial_ends_at)` só recebe plan e trial_ends_at — não recebe status da subscription. O @dev precisará ou (a) passar subscription.updated_at como parâmetro adicional, ou (b) consultar a tabela subscriptions na route antes de chamar getEffectiveTier, ou (c) expandir a assinatura do getEffectiveTier.
- **Recommendation:** Especificar no AC7 qual abordagem usar. A mais simples: na API route, após verificar o plano, fazer query adicional `supabase.from('subscriptions').select('status, updated_at').eq('user_id', userId).single()` e aplicar lógica de graça inline. Registrar como tech debt refatorar para getEffectiveTier futuramente.
