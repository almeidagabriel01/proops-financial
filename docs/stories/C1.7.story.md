## Context

PocketGuard (US) deve 4.8 estrelas no App Store a uma única feature: "In My Pocket" — um número simples que responde "quanto posso gastar hoje de forma segura?". É o item mais elogiado em reviews de PFM internacionalmente: "um número simples que me diz o que preciso saber".

O Finansim tem todos os dados necessários para calcular isso: receitas e despesas do mês (já exibidas nos `SummaryCards`) e as contas agendadas para o restante do mês (tabela `scheduled_transactions` com `status = 'pending'`).

A fórmula é simples:
```
saldo_mes = receitas_mes - despesas_mes
contas_pendentes = soma scheduled_transactions (pending, due_date até fim do mês)
safe_to_spend = saldo_mes - contas_pendentes
safe_to_spend_diario = safe_to_spend / dias_restantes_no_mes
```

O card fica abaixo dos `SummaryCards` existentes no dashboard — é a resposta direta à pergunta mais comum dos usuários. Disponível para Basic e Pro.

---

## User Story

> Como usuário,
> quero ver no dashboard quanto posso gastar hoje de forma segura,
> considerando meu saldo do mês e as contas que ainda tenho a pagar,
> para tomar decisões de consumo sem precisar fazer cálculos manualmente.

## Tech Notes for @dev

### Função de cálculo

```typescript
// src/lib/dashboard/safe-to-spend.ts

export interface SafeToSpendInput {
  totalIncome: number;
  totalExpenses: number;
  pendingBills: Array<{ amount: number; due_date: string }>;
  referenceDate?: Date;
}

export interface SafeToSpendResult {
  safeToSpend: number;
  safeToSpendDaily: number;
  balanceMonth: number;
  pendingTotal: number;
  daysRemaining: number;
  pendingBillsCount: number;
}

export function calculateSafeToSpend(input: SafeToSpendInput): SafeToSpendResult {
  const ref = input.referenceDate ?? new Date();
  const endOfMonth = new Date(ref.getFullYear(), ref.getMonth() + 1, 0); // último dia do mês
  const daysRemaining = Math.max(
    1,
    Math.ceil((endOfMonth.getTime() - ref.getTime()) / (1000 * 60 * 60 * 24)) + 1
  );

  // Filtrar apenas contas com due_date até o fim do mês atual
  const currentYearMonth = `${ref.getFullYear()}-${String(ref.getMonth() + 1).padStart(2, '0')}`;
  const relevantBills = input.pendingBills.filter(b =>
    b.due_date.startsWith(currentYearMonth) || b.due_date <= endOfMonth.toISOString().slice(0, 10)
  );

  const pendingTotal = relevantBills.reduce((sum, b) => sum + b.amount, 0);
  const balanceMonth = input.totalIncome - input.totalExpenses;
  const safeToSpend = balanceMonth - pendingTotal;
  const safeToSpendDaily = safeToSpend / daysRemaining;

  return {
    safeToSpend,
    safeToSpendDaily,
    balanceMonth,
    pendingTotal,
    daysRemaining,
    pendingBillsCount: relevantBills.length,
  };
}
```

#### MAJOR

**M1 — `new Date()` no servidor usa UTC; usuário brasileiro está em UTC-3**
`src: AC1 (função calculateSafeToSpend), AC2 (query dashboard)`

O servidor Vercel roda em UTC. `new Date()` no `dashboard/page.tsx` retorna hora UTC, não hora de Brasília. Entre 21h e 00h (BRT), o servidor já está no dia seguinte — `daysRemaining` e o cálculo do mês atual ficam incorretos. Às 23h30 de 30 de abril em Brasília, o servidor (em 02h30 UTC de 1 de maio) calcularia `daysRemaining` para maio com zero dados, mostrando `safeToSpend = 0`.

**Correção:** O dashboard já usa `?month=YYYY-MM` como URL param para controle de mês. A função `calculateSafeToSpend` deve receber a `referenceDate` derivada do param `month` do URL (ex: primeiro dia do mês selecionado como base), não de `new Date()`. Para `daysRemaining` no mês atual, usar o último dia do mês do param. O `dashboard/page.tsx` deve derivar `endOfMonth` a partir do param, não do relógio do servidor.

Exemplo: se `?month=2026-04`, `endOfMonth = 2026-04-30` e `daysRemaining` é calculado em relação ao dia atual do param (ou data de hoje em UTC-3, se disponível via header `x-vercel-ip-timezone`).

#### MINORs

**mn1 — Filtro `relevantBills` dentro da função é redundante**
A query de AC2 já filtra `due_date >= startOfToday AND due_date <= endOfMonth`. O filtro dentro de `calculateSafeToSpend` re-filtra os mesmos dados com lógica parcialmente equivalente (`b.due_date <= endOfMonth` seria verdadeiro para qualquer mês passado também). O filtro interno cria falsa impressão de segurança — remover ou simplificar para assert de consistência.

**mn2 — Distinção "sem dados" vs "saldo zero" na AC5**
AC5 usa `totalIncome = 0 AND totalExpenses = 0` para detectar "usuário novo sem dados". Mas `safeToSpend = 0` também pode ocorrer legitimamente (income exatamente igual a expenses + bills). O componente precisa receber um flag `hasData: boolean` separado (ex: `totalTransactions > 0`) para distinguir "sem importação" de "saldo zerado" — não pode depender apenas do valor de safeToSpend para decidir qual mensagem exibir.

**mn3 — Threshold de cor amarela (`safeToSpendDaily < R$20`) é valor absoluto arbitrário**
R$20/dia é um threshold fixo que não escala com o perfil financeiro do usuário. Um usuário com orçamento de R$500/mês tem safe-to-spend legítimo de ~R$16/dia. Para MVP é aceitável manter R$20, mas documentar que o threshold deve ser baseado em % do income médio em versões futuras.
