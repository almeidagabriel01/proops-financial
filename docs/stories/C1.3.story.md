## Context

O app é uma PWA instalável (`@serwist` configurado, `PwaInstallBanner` existe) mas não envia push notifications. O service worker atual (`src/sw.ts`) faz apenas precaching — não tem handler de eventos `push`.

Push notifications são pré-requisito para C1.4 (alertas de orçamento). Sem elas, o usuário só descobre que ultrapassou um orçamento quando ele mesmo abre o app — o que elimina o valor principal da feature de budgets.

Todos os concorrentes sérios (Mobills, Organizze, YNAB, Monarch Money) têm alertas de orçamento por push. É a "última milha" da feature de orçamento que já existe no app.

Esta story entrega apenas a infraestrutura: tabela de subscriptions, service worker com push handler, e as duas API routes necessárias. A lógica de _quando_ enviar push fica em C1.4.

---

## User Story

> Como usuário,
> quero autorizar o app a me enviar notificações push,
> para receber alertas financeiros importantes
> sem precisar abrir o app manualmente.

## Tech Notes for @dev

### Gerar chaves VAPID (executar uma vez, salvar no .env)

```bash
npx web-push generate-vapid-keys
# Saída:
# Public Key: BNxxxxxxx...
# Private Key: xxxxxxx...
```

Salvar no `.env.local` e nas variáveis de ambiente do Vercel (Settings → Environment Variables).

### Configurar web-push no servidor

```typescript
// src/lib/push/config.ts
import webpush from 'web-push';

webpush.setVapidDetails(
  process.env.VAPID_SUBJECT!,
  process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY!,
  process.env.VAPID_PRIVATE_KEY!,
);

export { webpush };
```

### Subscribe no frontend

```typescript
// src/lib/push/subscribe.ts
export async function subscribeToPush(): Promise<boolean> {
  if (!('serviceWorker' in navigator) || !('PushManager' in window)) return false;

  const registration = await navigator.serviceWorker.ready;
  const subscription = await registration.pushManager.subscribe({
    userVisibleOnly: true,
    applicationServerKey: process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY,
  });

  const { endpoint, keys } = subscription.toJSON() as {
    endpoint: string;
    keys: { p256dh: string; auth: string };
  };

  const res = await fetch('/api/push/subscribe', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ endpoint, keys }),
  });

  return res.ok;
}
```

### Handler de push no service worker (src/sw.ts)

O `src/sw.ts` usa Serwist — adicionar os listeners antes de `serwist.addEventListeners()`:

```typescript
self.addEventListener('push', (event: PushEvent) => {
  const data = event.data?.json() ?? {};
  event.waitUntil(
    self.registration.showNotification(data.title ?? 'Finansim', {
      body: data.body ?? '',
      icon: '/icons/icon-192x192.png',
      data: { url: data.url ?? '/dashboard' },
    })
  );
});

self.addEventListener('notificationclick', (event: NotificationEvent) => {
  event.notification.close();
  event.waitUntil(
    clients.matchAll({ type: 'window' }).then((clientList) => {
      const url = event.notification.data?.url ?? '/dashboard';
      const existing = clientList.find((c) => c.url === url && 'focus' in c);
      if (existing) return existing.focus();
      return clients.openWindow(url);
    })
  );
});
```

### API /api/push/send — validação interna

```typescript
const authHeader = request.headers.get('Authorization');
const expectedToken = `Bearer ${process.env.SUPABASE_SERVICE_ROLE_KEY}`;
```

#### MAJOR

**M1 — Inconsistência crítica no nome da env var VAPID (causa bug silencioso)**
`src: AC2 (.env.example) vs. Tech Notes (código)`

AC2 documenta `.env.example` com `VAPID_PUBLIC_KEY=...`, mas o código do `subscribeToPush()` em Tech Notes usa `process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY`. O frontend Next.js só acessa variáveis com o prefixo `NEXT_PUBLIC_` — sem ele, o valor é `undefined` no browser. Resultado: `pushManager.subscribe({ applicationServerKey: undefined })` falha com `DOMException: applicationServerKey must be a valid VAPID key`, e o usuário não recebe pushes, sem erro visível no app.

**Correção necessária em AC2:** o `.env.example` deve ter ambas as entradas:
```
NEXT_PUBLIC_VAPID_PUBLIC_KEY=your-vapid-public-key  # usado pelo frontend
VAPID_PRIVATE_KEY=your-vapid-private-key             # servidor apenas
VAPID_SUBJECT=mailto:suporte@finansim.app
```

#### MINORs

**mn1 — Rotação de VAPID keys não documentada (risco operacional)**
VAPID keys são geradas uma vez e permanentes. Se o `VAPID_PRIVATE_KEY` vazar, as keys precisam ser rotacionadas — mas isso invalida TODAS as subscriptions existentes (browsers rejeitam pushs com as novas keys). Adicionar aos tech notes: "VAPID keys são de longa duração. Rotacionar exige campanha de re-subscribe: o banner `PushPermissionBanner` deve ser re-exibido para usuários existentes após rotação."

**mn2 — TypeScript types de PushEvent/NotificationEvent em sw.ts**
`src/sw.ts` usa `SerwistGlobalConfig` do `@serwist/next` mas `PushEvent` e `NotificationEvent` podem não estar no escopo. Verificar se o `tsconfig` do service worker inclui `"lib": ["WebWorker"]` ou se é necessário adicionar `/// <reference lib="webworker" />` no topo do arquivo para evitar erro de compilação.

#### MAJORs

**M1 — `/api/push/send` usa cliente errado do Supabase — push nunca entregue**
`src: src/app/api/push/send/route.ts:38`

A rota usa `createClient()` (anon key + cookie auth). Como é chamada pela Edge Function `check-budget-alerts` sem sessão de usuário, `auth.uid()` retorna `null`. A política RLS `USING (user_id = auth.uid())` bloqueia silenciosamente o SELECT em `push_subscriptions` — o array de subscriptions retorna vazio, a rota sempre responde `{ sent: 0, failed: 0 }`, e nenhuma notificação é enviada.

**Fix:** Substituir `createClient()` por `createServiceClient()` (já existe em `@/lib/supabase/server` — usa `SUPABASE_SERVICE_ROLE_KEY`, bypass de RLS) apenas na rota de send. A rota de subscribe continua com `createClient()` (opera com sessão de usuário).

```typescript
// src/app/api/push/send/route.ts — linha 3
import { createServiceClient } from '@/lib/supabase/server';
// ...
const supabase = await createServiceClient(); // linha 38
```

**M2 — Migration sem política UPDATE — upsert falha em re-subscribe**
`src: supabase/migrations/019_push_subscriptions.sql`

A rota de subscribe usa `.upsert({ onConflict: 'user_id,endpoint' })`, que gera `INSERT ... ON CONFLICT DO UPDATE`. O PostgreSQL verifica políticas RLS de UPDATE para o path ON CONFLICT. Sem política UPDATE, o path de update é bloqueado por RLS — re-subscribe (mesmo endpoint, keys atualizadas) falha. Apenas o primeiro subscribe funciona; qualquer atualização de subscription subsequente silencia com erro RLS.

**Fix:** Nova migration adicionando a política UPDATE:

```sql
-- supabase/migrations/020_push_subscriptions_update_policy.sql
CREATE POLICY "push_subscriptions_update_own"
  ON public.push_subscriptions
  FOR UPDATE
  USING (user_id = auth.uid())
  WITH CHECK (user_id = auth.uid());
```

#### MINORs

**mn1 — Correspondência parcial de URL no `notificationclick` pode focar janela errada**
`src: src/sw.ts`

`clientList.find((c) => c.url.includes(url))` — se `url` for `/dashboard` e outra aba tiver `/dashboard/analytics` aberta, a aba errada recebe o foco. Usar comparação de URL completa:

```typescript
const existing = clientList.find(
  (c) => new URL(c.url).pathname === url && 'focus' in c
);
```

**mn2 — `setRequesting(false)` ausente no bloco `finally`**
`src: src/components/push/push-permission-banner.tsx:27-37`

`handleActivate()` define `setRequesting(true)` no início mas nunca reseta para `false`. O `finally` só chama `setVisible(false)`. Inócuo enquanto o banner desaparece no finally, mas se a lógica de visibilidade mudar futuramente, os botões ficam permanentemente desabilitados.

```typescript
} finally {
  setRequesting(false); // adicionar antes de setVisible
  setVisible(false);
}
```
