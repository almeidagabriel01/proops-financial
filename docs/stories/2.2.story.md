## Context

A IA categoriza bem, mas nao e perfeita. Esta story fecha o loop de aprendizado: o usuario corrige categorizacoes erradas, e o sistema aprende para nunca repetir o mesmo erro. O aprendizado e persistido na tabela `category_dictionary` por usuario.

A hierarquia de categorizacao (arch section 5.1) ja preve esta camada:
1. **Dicionario do usuario** (`category_dictionary`) — esta story popula esta camada
2. Cache global (`category_cache`) — Story 2.1
3. Claude Haiku — Story 2.1

Apos uma correcao, todas as transacoes futuras com a mesma descricao normalizada usam o dicionario do usuario — sem chamar a IA. Transacoes passadas com a mesma descricao devem ser atualizadas em batch opcional ("corrigir todas").

**PRD refs:** FR8 (corrigir + aprender), FR9 (dicionario que melhora com uso)
**Arch refs:** `category_dictionary` table (section 3.1), Tier 1 hierarchy (section 5.1)

## User Story

> Como usuario,
> quero poder corrigir a categoria de uma transacao,
> para que o app aprenda minhas preferencias e melhore ao longo do tempo.

## Acceptance Criteria

### AC1: Detalhe da transacao com opcao de corrigir categoria
- [ ] Na lista de transacoes (`/transactions`), tap/click em uma transacao abre painel de detalhe (Sheet ou Dialog do shadcn/ui)
- [ ] Painel exibe: data, descricao completa, valor formatado em BRL, categoria atual com icone/cor
- [ ] Botao "Corrigir categoria" visivel no painel
- [ ] **Verificacao:** Tap em qualquer transacao; painel abre com dados corretos; botao de correcao visivel

### AC2: Selector de categoria
- [ ] `src/components/transactions/category-selector.tsx` — componente reutilizavel
- [ ] Exibe as 14 categorias fixas em grid com icone + nome (PT-BR)
- [ ] Categoria atual destacada visualmente (cor diferente / borda)
- [ ] Icones por categoria (Lucide Icons): alimentacao=`Utensils`, delivery=`Package`, transporte=`Car`, moradia=`Home`, saude=`Heart`, educacao=`BookOpen`, lazer=`Music`, compras=`ShoppingBag`, assinaturas=`Repeat`, transferencias=`ArrowLeftRight`, salario=`Banknote`, investimentos=`TrendingUp`, impostos=`Receipt`, outros=`MoreHorizontal`
- [ ] **Verificacao:** Selector exibe 14 categorias com icones; categoria atual esta destacada

### AC3: Salvar correcao e atualizar dicionario
- [ ] Ao selecionar nova categoria: `UPDATE transactions SET category=$1, category_source='user', updated_at=now() WHERE id=$2`
- [ ] Upsert em `category_dictionary`: `INSERT INTO category_dictionary (user_id, description_pattern, category) VALUES ($1, normalize_description($description), $3) ON CONFLICT (user_id, description_pattern) DO UPDATE SET category=$3, usage_count=usage_count+1, updated_at=now()`
- [ ] Feedback visual imediato: categoria atualizada na lista sem reload de pagina
- [ ] Painel fecha apos salvar com sucesso
- [ ] Tratamento de erro: mensagem clara em PT-BR se falhar
- [ ] **Verificacao:** Corrigir categoria de uma transacao; dicionario atualizado no banco; lista mostra nova categoria imediatamente

### AC4: Correcao em batch — "corrigir todas"
- [ ] Apos selecionar nova categoria, mostrar opcao: "Corrigir so esta" vs "Corrigir todas com a mesma descricao (N transacoes)"
- [ ] "Corrigir todas": `UPDATE transactions SET category=$1, category_source='user' WHERE user_id=$2 AND id = ANY($ids_com_mesma_descricao)`
- [ ] Calcular `ids_com_mesma_descricao` via `normalize_description` antes de mostrar a opcao
- [ ] Se houver apenas 1 transacao com aquela descricao: nao mostrar a opcao batch (simplificacao UX)
- [ ] **Verificacao:** Extrato com 5 transacoes "UBER": corrigir uma com "Corrigir todas"; todas as 5 mudam de categoria

### AC5: Hierarquia de categorizacao respeitada em novas importacoes
- [ ] A Edge Function `categorize-import` (Story 2.1 AC3) JA implementa Tier 1 (dicionario do usuario)
- [ ] Verificar que nova importacao com descricao ja corrigida usa `category_source='user'` e nao chama a IA
- [ ] **Verificacao:** Corrigir "UBER" para `transporte`; importar novo extrato com transacoes "UBER"; todas tem `category_source='user'` sem chamar Haiku

### AC6: Persistencia no banco com RLS
- [ ] Correcao de categoria: usuario so pode atualizar suas proprias transacoes (RLS garante)
- [ ] `category_dictionary`: usuario so pode CRUD seu proprio dicionario (RLS conforme schema)
- [ ] Nao usar service role para estas operacoes — usar browser client com RLS
- [ ] **Verificacao:** Tentar atualizar transacao de outro usuario via API; deve retornar 0 rows updated

### AC7: Testes unitarios
- [ ] `tests/unit/ai/category-correction.test.ts` — testar logica de normalizacao e upsert no dicionario
- [ ] Testar batch correction: N transacoes com mesma descricao sao todas atualizadas
- [ ] Testar que correcao unica nao afeta outras transacoes
- [ ] **Verificacao:** `npm test` passa sem erros

## Tech Notes for @dev

### Normalizacao de descricao (client-side)
```typescript
// src/lib/utils/format.ts — adicionar funcao:
export function normalizeDescription(raw: string): string {
  return raw
    .toLowerCase()
    .normalize('NFD').replace(/[\u0300-\u036f]/g, '')
    .replace(/[^a-z0-9 ]/g, '')
    .replace(/\s+/g, ' ')
    .trim()
}
// IMPORTANTE: deve ser identica a public.normalize_description() SQL
```

### Upsert no dicionario via Supabase client
```typescript
const supabase = createClientBrowser() // client-side, RLS aplicado
await supabase
  .from('category_dictionary')
  .upsert({
    user_id: userId,
    description_pattern: normalizeDescription(transaction.description),
    category: newCategory,
    usage_count: 1
  }, { onConflict: 'user_id,description_pattern' })
```

### Buscar transacoes com mesma descricao normalizada

const pattern = normalizeDescription(transaction.description)
const { data: sameDesc } = await supabase
