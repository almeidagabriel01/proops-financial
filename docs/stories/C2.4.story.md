## Context

Todos os concorrentes sérios enviam um resumo mensal: Mobills, Organizze, Monarch Money, Copilot Money. É o touchpoint de retenção mais poderoso — chega no início do mês quando o usuário ainda está pensando no mês anterior, e leva o app de volta à mente sem exigir nenhuma ação.

O stack está pronto:
- **Vercel Cron**: agendamento no D+1 de cada mês (grátis no plano Hobby, incluído no Pro)
- **Resend**: envio de email com PDF em attachment — 3.000 emails/mês grátis
- **@react-pdf/renderer**: geração de PDF server-side em Next.js (sem browser/Puppeteer)
- **Supabase Storage**: armazenar o PDF gerado para download posterior em `/settings`

O relatório cobre o mês anterior completo (dados fechados) e é gerado de madrugada para evitar carga no horário de pico. O usuário pode optar por não receber (opt-out em `/settings`).

**Conteúdo do relatório PDF:**
1. Cabeçalho: "Seu relatório financeiro — [Mês Ano]"
2. Resumo: receitas totais, despesas totais, saldo (receita − despesa), taxa de poupança
3. Score de saúde: badge e número (requer C2.3 implementada)
4. Top 5 categorias de gasto com valor e % do total
5. Comparativo com mês anterior: receitas, despesas, saldo (↑↓ com percentual)
6. Status de orçamentos: quantos cumpridos vs estourados
7. Status de metas: metas ativas e % de progresso de cada uma
8. Rodapé: "Gerado pelo Finansim em [data]"

---

## User Story

> Como usuário,
> quero receber automaticamente no início de cada mês um relatório em PDF com o resumo do mês anterior,
> para revisar meu desempenho financeiro sem precisar abrir o app,
> e ter esse relatório disponível para download quando precisar.

### AC1: Migration — tabela `monthly_reports`

- [ ] Nova migration `025_monthly_reports.sql` em `supabase/migrations/`
- [ ] Schema da tabela:
  ```sql
  CREATE TABLE monthly_reports (
    id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
    month date NOT NULL,   -- primeiro dia do mês coberto: 2026-03-01
    status text NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'generating', 'completed', 'failed')),
    pdf_storage_path text,
    pdf_url text,
    email_sent_at timestamptz,
    error_message text,
    created_at timestamptz NOT NULL DEFAULT now(),
    updated_at timestamptz NOT NULL DEFAULT now(),
    UNIQUE(user_id, month)
  );
  ```
- [ ] RLS habilitado: SELECT por `user_id = auth.uid()`; INSERT/UPDATE apenas pelo service role (geração server-side)
- [ ] Trigger `update_updated_at()` na tabela
- [ ] **Verificação:** `supabase db push` sem erros

### AC2: Opt-out em `/settings`

- [ ] Nova preferência de usuário `monthly_report_email` (boolean, **default `false`** — opt-IN, nunca opt-out) na tabela `profiles`
- [ ] Migration adicionada como parte de `025_monthly_reports.sql`
- [ ] UI em `/settings` (aba "Perfil" ou nova aba "Notificações"): **checkbox desmarcado por padrão** com label exato: "Quero receber meu resumo financeiro mensal por email"
- [ ] Ao marcar o checkbox: persistir `monthly_report_email = true` via `PATCH /api/user/preferences`
- [ ] Ao desmarcar: persistir `monthly_report_email = false`
- [ ] O Cron **não envia email** para usuários com `monthly_report_email = false` (default)
- [ ] **LGPD:** o consentimento é opt-IN explícito — nenhum email é enviado sem ação afirmativa do usuário
- [ ] **Verificação:** novo usuário → checkbox desmarcado; marcar → email enviado no próximo cron; desmarcar → não recebe mais

### AC3: Endpoint de geração — `POST /api/reports/monthly/generate`

- [ ] Aceita body: `{ userId: string, month: string }` (chamado pelo Cron)
- [ ] Protegido por header `Authorization: Bearer ${CRON_SECRET}` (variável de ambiente)
- [ ] Fluxo de geração:
  1. Verificar se relatório para o mês/usuário já existe e está `completed` → pular
  2. Inserir registro em `monthly_reports` com `status = 'generating'`
  3. Buscar dados financeiros do mês: transações, totais por categoria, orçamentos, metas, score de saúde (reusar lógica de C2.3)
  4. Gerar PDF com `@react-pdf/renderer` (ver Tech Notes)
  5. Fazer upload do PDF para Supabase Storage em `reports/{userId}/{month}.pdf`
  6. Atualizar `monthly_reports`: `status = 'completed'`, `pdf_storage_path`, `pdf_url`
  7. Se `monthly_report_email = true`: enviar email via Resend com PDF em attachment
  8. Atualizar `email_sent_at`
- [ ] Em caso de erro em qualquer step: `status = 'failed'`, `error_message` com detalhes
- [ ] **Verificação:** chamar endpoint com userId e month válidos → PDF gerado e armazenado; status = 'completed'

### AC4: Vercel Cron — execução automática D+1 às 6h

- [ ] Configurar cron em `vercel.json`:
  ```json
  {
    "crons": [{
      "path": "/api/reports/monthly/cron",
      "schedule": "0 9 1 * *"
    }]
  }
  ```
  (9h UTC = 6h BRT — meia-noite de Brasília é UTC-3, ajustar conforme DST)
- [ ] Endpoint `GET /api/reports/monthly/cron`:
  - Protegido por header `Authorization: Bearer ${CRON_SECRET}` (Vercel injeta automaticamente)
  - Busca todos os usuários com pelo menos 1 transação no mês anterior
  - Para cada usuário: chama a lógica de geração (AC3) em loop sequencial ou em paralelo com `Promise.allSettled`
  - Loga resultado: `{ total, succeeded, failed }`
- [ ] Variáveis de ambiente necessárias: `CRON_SECRET`, `RESEND_API_KEY`, `NEXT_PUBLIC_SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`
- [ ] **Verificação:** acionar cron manualmente com header correto → relatórios gerados para usuários elegíveis

### AC5: Download do relatório em `/settings`

- [ ] Aba "Dados" em `/settings` exibe seção "Relatórios Mensais"
- [ ] Lista os últimos 12 relatórios gerados com: mês, status, botão "Baixar PDF" (abre URL assinada do Supabase Storage)
- [ ] API `GET /api/reports/monthly` retorna a lista de relatórios do usuário
- [ ] API `GET /api/reports/monthly/[month]/download` retorna uma URL assinada (signed URL, expira em 1h) para download seguro
- [ ] Se nenhum relatório gerado ainda: exibe "Seu primeiro relatório será gerado no dia 1 do próximo mês."
- [ ] **Verificação:** após geração, entrar em /settings → relatório listado; clicar "Baixar PDF" → download inicia

### AC6: Email via Resend

- [ ] Integrar Resend SDK: `npm install resend`
- [ ] Email enviado de: `relatorio@finansim.com.br` (ou domínio configurado no Resend)
- [ ] Assunto: "Seu relatório financeiro de [Mês Ano] está pronto, [nome_do_usuário]!"

## Pré-requisitos Operacionais

> ⚠️ **Sem esses passos, os emails cairão em spam ou serão rejeitados pelo servidor de destino.** Executar antes de ativar o Cron em produção.

### Checklist de configuração Resend + Domínio

- [ ] **1. Criar conta no Resend** — [resend.com](https://resend.com). Plano gratuito: 3.000 emails/mês, suficiente para MVP.
- [ ] **2. Adicionar domínio no Resend** — painel Resend → "Domains" → "Add Domain" → inserir o domínio do Finansim.
- [ ] **3. Configurar registros DNS** — o Resend exibirá os registros a adicionar no provedor DNS (Cloudflare, Registro.br, etc.):
  - **SPF:** registro TXT em `@` ou no subdomínio — ex: `v=spf1 include:amazonses.com ~all`
  - **DKIM:** registros CNAME gerados pelo Resend (geralmente 3 entradas)
  - **DMARC (recomendado):** `_dmarc.seudominio.com TXT "v=DMARC1; p=none; rua=mailto:dmarc@seudominio.com"`
- [ ] **4. Verificar domínio no Resend** — após propagação DNS (até 48h), clicar "Verify" no painel. Status deve ficar "Verified".
- [ ] **5. Gerar API Key** — Resend → "API Keys" → "Create API Key" com permissão `Sending access`.
- [ ] **6. Adicionar variáveis de ambiente:**
  ```bash
  # .env.local (desenvolvimento)
  RESEND_API_KEY=re_xxxxxxxxxxxx

  # Vercel Dashboard → Settings → Environment Variables (produção)
  RESEND_API_KEY=re_xxxxxxxxxxxx
  ```
- [ ] **7. Atualizar `.env.example`:**
  ```bash
  # Resend — envio de emails transacionais
  # Obter em: resend.com → API Keys
  RESEND_API_KEY=
  ```
- [ ] **8. Configurar remetente** — usar endereço no domínio verificado: `noreply@seudominio.com` ou `relatorio@seudominio.com`. Nunca usar domínio não verificado.

## Tech Notes for @dev

### Migration

```sql
-- supabase/migrations/025_monthly_reports.sql

-- Tabela de relatórios
CREATE TABLE monthly_reports (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  month date NOT NULL,
  status text NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'generating', 'completed', 'failed')),
  pdf_storage_path text,
  pdf_url text,
  email_sent_at timestamptz,
  error_message text,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE(user_id, month)
);

ALTER TABLE monthly_reports ENABLE ROW LEVEL SECURITY;

-- Usuários podem ver seus próprios relatórios
CREATE POLICY "users can view their own reports"
  ON monthly_reports FOR SELECT
  USING (user_id = auth.uid());

-- Apenas service role pode inserir e atualizar (geração server-side)
CREATE POLICY "service role can manage reports"
  ON monthly_reports FOR ALL
  USING (auth.role() = 'service_role');

CREATE TRIGGER update_monthly_reports_updated_at
  BEFORE UPDATE ON monthly_reports
  FOR EACH ROW EXECUTE FUNCTION update_updated_at();

-- Adicionar preferência de opt-IN na tabela profiles
-- DEFAULT false: LGPD — consentimento explícito obrigatório, nunca inscrever por padrão
ALTER TABLE profiles ADD COLUMN IF NOT EXISTS monthly_report_email boolean NOT NULL DEFAULT false;
```

### Resend integration

```typescript
// src/lib/email/send-monthly-report.ts
import { Resend } from 'resend';

const resend = new Resend(process.env.RESEND_API_KEY);

export async function sendMonthlyReportEmail(
  email: string,
  name: string,
  month: string,
  pdfBuffer: Buffer
) {
  await resend.emails.send({
    from: 'Finansim <relatorio@finansim.com.br>',

    attachments: [{
      filename: `relatorio-${month}.pdf`,
      content: pdfBuffer,
    }],
  });
}
```

## Out of Scope

- Relatório em tempo real (sob demanda pelo usuário além do download — gerar apenas no cron)
- Personalização do conteúdo do relatório pelo usuário
- Envio por WhatsApp
- Relatório trimestral ou anual (apenas mensal no MVP)
- Internacionalização do PDF (apenas PT-BR)
- Relatório para usuários sem nenhuma transação no mês (não gerar — sem dados, sem relatório)
- Retry automático de emails falhados (apenas log de erro no MVP)

### Revisão Pré-Implementação — 2026-04-14
**Agente:** Quinn (@qa)

**Gate:** 1 BLOCKER · 1 MAJOR · 1 MINOR

**NÃO autorizado para desenvolvimento. Corrigir BLOCKER e MAJOR antes de retornar ao @sm para atualização da story.** ❌

#### BLOCKER

**bl1 — LGPD: opt-out por padrão viola consentimento explícito**
`monthly_report_email DEFAULT true` inscreve o usuário para receber dados financeiros por email sem consentimento prévio. O CLAUDE.md diz: "LGPD desde o dia 1 — consentimento explícito". O Art. 7, inciso I da LGPD exige consentimento livre, informado e inequívoco para tratamento de dados pessoais — envio de resumo financeiro por email é tratamento de dados sensíveis.

**Correção obrigatória (uma das opções):**
- **Opção A (preferida):** Alterar `DEFAULT false`. Adicionar toggle de opt-IN em `/settings` com linguagem clara: "Quero receber meu resumo financeiro mensal por email." Não enviar emails até que o usuário ative explicitamente.
- **Opção B:** Coletar consentimento explícito no onboarding (nova etapa) com checkbox não pré-marcado.

O campo de opt-out é positivo — mas deve ser opt-IN, não opt-OUT.

#### MAJOR

**mj1 — Domínio Resend não documentado como pré-requisito**
A story usa `relatorio@finansim.com.br` mas não captura como pré-requisito:
1. Verificação do domínio `finansim.com.br` no painel Resend (DNS TXT record de verificação)
2. Configuração de registros SPF e DKIM no DNS do domínio
Sem isso, 100% dos emails irão para spam/quarentena. A feature estará "funcionando" tecnicamente mas inoperante na prática. Adicionar na story como pré-requisito operacional com checklist:
- [ ] Domínio verificado no Resend
- [ ] SPF configurado: `v=spf1 include:amazonses.com ~all` (Resend usa SES)
- [ ] DKIM: chaves geradas e configuradas no DNS

#### MINOR

**mn1 — Falha silenciosa para o usuário**
`status = 'failed'` em `/settings` é visível, mas sem ação disponível (sem "Tentar novamente", sem notificação ativa). Para MVP, documentar explicitamente na story que a experiência de falha é passiva (usuário vê status "failed", sem botão de retry). Evita que o @dev deixe em aberto ou implemente comportamento diferente.
