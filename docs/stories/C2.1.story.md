## Context

A IA categoriza bem, mas o usuário não tem como criar regras explícitas e permanentes. Quando o usuário corrige "iFood" de `outros` para `delivery`, a correção é salva no `category_dictionary` (pattern aprendido), mas o usuário não sabe disso — e não consegue criar regras novas ou gerenciar as existentes.

A tabela `category_dictionary` (user_id, description_pattern, category, usage_count) já existe e é usada pelo sistema de aprendizado. O Motor de Regras é uma camada **Tier 0** que roda **antes** da IA no fluxo de importação: primeiro verifica regras manuais, depois o cache global, depois a IA. Isso economiza tokens, melhora consistência e dá controle ao usuário.

Copilot Money é o líder global com esse padrão ("Rules Engine"). É citado em reviews como a razão principal de lealdade ao produto: "finalmente um app que não precisa de babysitting". Nenhum concorrente brasileiro oferece isso.

A feature tem dois componentes:
1. **UI de gerenciamento** — `/more/regras` onde o usuário cria, edita, reordena e desativa regras
2. **Integração no fluxo de import** — as regras são verificadas antes da IA

Bonus opcional: quando o usuário corrige manualmente uma categoria, oferecer "Criar regra automática para '{{descrição}}'?".

## User Story

> Como usuário,
> quero criar regras explícitas de categorização como "se a descrição contém 'iFood' → sempre categorizar como Delivery",
> para que minhas transações recorrentes sejam sempre categorizadas corretamente sem que eu precise corrigi-las toda vez.

CREATE TABLE categorization_rules (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  pattern text NOT NULL CHECK (char_length(pattern) >= 2),
  match_type text NOT NULL DEFAULT 'contains' CHECK (match_type IN ('contains', 'exact', 'starts_with')),
  category text NOT NULL,
  priority integer NOT NULL DEFAULT 0,
  active boolean NOT NULL DEFAULT true,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()

ALTER TABLE categorization_rules ENABLE ROW LEVEL SECURITY;

CREATE POLICY "users can manage their own rules"
  ON categorization_rules FOR ALL
  USING (user_id = auth.uid())

CREATE INDEX idx_categorization_rules_lookup
  ON categorization_rules(user_id, active, priority DESC);

  BEFORE UPDATE ON categorization_rules

### Função de aplicação de regras

```typescript
// src/lib/categorization/apply-rules.ts
import { createClient } from '@/lib/supabase/server';

export async function applyCategorizationRules(
  userId: string,
  description: string
): Promise<{ category: string; ruleId: string } | null> {
  const supabase = createClient();
  
  const { data: rules } = await supabase
    .from('categorization_rules')
    .select('id, pattern, match_type, category')
    .eq('user_id', userId)
    .eq('active', true)
    .order('priority', { ascending: false });

  if (!rules?.length) return null;

  const desc = description.toLowerCase();
  
  for (const rule of rules) {
    const pattern = rule.pattern.toLowerCase();
    const matched =
      rule.match_type === 'contains' ? desc.includes(pattern) :
      rule.match_type === 'exact' ? desc === pattern :
      rule.match_type === 'starts_with' ? desc.startsWith(pattern) :
      false;
    
    if (matched) return { category: rule.category, ruleId: rule.id };
  }
  
  return null;
}
```

## Out of Scope

- Regex como tipo de match (apenas contains/exact/starts_with no MVP)
- Importação/exportação de regras em CSV
- Regras globais (compartilhadas entre usuários)
- Regras baseadas em valor (ex: "se valor > R$500 → categoria X")
- Regras com múltiplas condições (AND/OR)
- Reordenação drag-and-drop da prioridade (campo numérico é suficiente no MVP)

#### MAJOR

**mj1 — Tier ordering não define posição do `category_dictionary`**
A story diz que as regras rodam "antes do cache e da IA", mas o projeto tem dois sistemas pré-IA: `category_cache` (cache global) e `category_dictionary` (padrões aprendidos do usuário). A AC4 deve documentar a ordem completa de lookup:
```
Tier 0: categorization_rules  → category_source = 'rule'
Tier 1: category_dictionary   → category_source = 'user' (existente)
Tier 2: category_cache        → category_source = 'cache' (existente)
Tier 3: IA                    → category_source = 'ai' (existente)
```
Se não for explícito, o @dev pode inserir as regras após o dictionary, quebrando a intenção de "regras explícitas têm prioridade absoluta".

#### MINOR

**mn1 — Sem normalização de acentos no pattern matching**
O Tech Notes usa `.toLowerCase()` mas descrições bancárias BR têm variações com/sem acento ("IFOOD", "Ífood"). A migration `002_unaccent_search.sql` já instalou a extensão `unaccent` no banco. O @dev deve aplicar `unaccent()` ao comparar no banco, ou normalizar via `normalize('NFD', str).replace(/\p{Mn}/gu, '')` no JS antes do match.
