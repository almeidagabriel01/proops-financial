## Context

A feature de orçamentos já existe e funciona bem — `GET /api/budgets` calcula o gasto real vs limite para cada orçamento ativo. Mas o usuário só descobre que ultrapassou um limite se ele mesmo abrir o app e navegar até `/more/orcamentos`.

Todos os concorrentes sérios têm alertas de orçamento: Mobills, Organizze, YNAB, Monarch Money, PocketGuard. É a feature mais solicitada em reviews de usuários de apps financeiros: "perco o controle perto do final do mês".

Esta story adiciona a camada de notificação proativa: uma Supabase Edge Function que roda periodicamente, verifica os orçamentos de todos os usuários com push subscription ativa, e envia notificações nos thresholds de 80% e 100%. Uma tabela `budget_alert_log` evita spam de notificações (máximo uma notificação por threshold por orçamento por mês).

**Pré-requisito:** C1.3 (push subscriptions) deve estar implementada e deployada antes desta story.

---

## User Story

> Como usuário com orçamento configurado,
> quero receber uma notificação push quando atingir 80% e 100% do limite de uma categoria,
> para agir antes de estourar o orçamento
> sem precisar verificar o app manualmente.

## Acceptance Criteria

### AC1: Migration — tabela `budget_alert_log`

- [ ] Nova migration `017_budget_alert_log.sql` em `supabase/migrations/`
- [ ] Schema:
  ```sql
  CREATE TABLE budget_alert_log (
    id          uuid DEFAULT gen_random_uuid() PRIMARY KEY,
    user_id     uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
    budget_id   uuid NOT NULL REFERENCES budgets(id) ON DELETE CASCADE,
    threshold   int NOT NULL CHECK (threshold IN (80, 100)),
    month       text NOT NULL,  -- formato YYYY-MM
    sent_at     timestamptz DEFAULT now(),
    CONSTRAINT budget_alert_log_unique UNIQUE(budget_id, threshold, month)
  );
  ```
- [ ] RLS: `user_id = auth.uid()` para SELECT (Edge Function usa service role — não precisa de INSERT/DELETE via RLS)
- [ ] Índice em `(user_id, month)` para queries eficientes
- [ ] **Verificação:** `supabase db push` sem erros

### AC2: Edge Function `check-budget-alerts`

- [ ] Criada em `supabase/functions/check-budget-alerts/index.ts`
- [ ] Usa Supabase service role (via env `SUPABASE_SERVICE_ROLE_KEY`) para acessar dados de todos os usuários
- [ ] Lógica de execução:
  1. Buscar todos os orçamentos com `status = 'active'` (ou todos — tabela `budgets` não tem campo status, usar todos)
  2. Para cada orçamento, calcular o gasto real do mês atual:
     ```sql
     SELECT COALESCE(SUM(amount), 0) as spent
     FROM transactions
     WHERE user_id = $userId
       AND category = $category
       AND type = 'debit'
       AND date >= date_trunc('month', now())
       AND date < date_trunc('month', now()) + interval '1 month'
     ```
  3. Calcular percentual: `pct = (spent / monthly_limit) * 100`
  4. Para cada threshold (80, 100) que for atingido:
     - Verificar se já existe entrada em `budget_alert_log` com `budget_id + threshold + month_atual`
     - Se não existe: enviar push via `POST /api/push/send` (interno) e inserir log
- [ ] Mensagens das notificações:
  - 80%: `"Você usou 80% do orçamento de [categoria] este mês (R$ X,XX de R$ Y,YY)"`
  - 100%: `"Orçamento de [categoria] estourado! (R$ X,XX gastos, limite R$ Y,YY)"`
  - URL da notificação: `/more/orcamentos`
- [ ] Se `POST /api/push/send` falhar: logar erro, continuar para próximo orçamento (não interromper)
- [ ] **Verificação:** Executar a Edge Function manualmente com `supabase functions invoke check-budget-alerts` → logs mostram orçamentos processados e pushs enviados

### AC3: Agendamento via pg_cron (hourly)

- [ ] Adicionar configuração de cron à migration `017_budget_alert_log.sql` (ou migration separada):
  ```sql
  SELECT cron.schedule(
    'check-budget-alerts-hourly',
    '0 * * * *',  -- toda hora
    $$
    SELECT net.http_post(
      url := current_setting('app.edge_functions_url') || '/check-budget-alerts',
      headers := '{"Authorization": "Bearer ' || current_setting('app.service_role_key') || '"}'::jsonb
    );
    $$
  );
  ```
- [ ] Alternativa se pg_cron não disponível: documentar no README da Edge Function como configurar via Supabase Dashboard → Database → Cron Jobs
- [ ] **Verificação:** Cron job listado em Supabase Dashboard → Database → Cron Jobs

### AC4: Trigger por import (opcional, se pg_cron não disponível)

- [ ] Como alternativa ao cron horário, `POST /api/import` chama `check-budget-alerts` após completar a categorização:
  ```typescript
  // Após salvar transações, em background:
  fetch('/api/push/check-budgets', { method: 'POST' })
    .catch(() => {}); // best-effort, não bloqueia resposta
  ```
- [ ] `POST /api/push/check-budgets` — rota interna que invoca a Edge Function via service role
- [ ] Esta rota é adicional ao cron — não substitui, apenas complementa
- [ ] **Verificação:** Importar extrato com despesas que ultrapassam orçamento → push recebido em até 1 minuto

### AC5: Graceful degradation sem push subscription

- [ ] Se o usuário não tem subscription ativa em `push_subscriptions`: Edge Function pula silenciosamente (sem erro)
- [ ] Alertas são calculados mesmo assim (para quando o usuário ativar push no futuro, o log evita re-envio retroativo de alertas do mês)
- [ ] **Verificação:** Rodar Edge Function com usuário sem subscription → sem erros nos logs, `sent: 0`

## Tech Notes for @dev

### Estrutura da Edge Function

```typescript
// supabase/functions/check-budget-alerts/index.ts
import { serve } from 'https://deno.land/std@0.177.0/http/server.ts';
import { createClient } from 'https://esm.sh/@supabase/supabase-js@2';

serve(async () => {
  const supabase = createClient(
    Deno.env.get('SUPABASE_URL')!,
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!,
  );

  const currentMonth = new Date().toISOString().slice(0, 7); // YYYY-MM

  // 1. Buscar todos os orçamentos
  const { data: budgets } = await supabase
    .from('budgets')
    .select('id, user_id, category, monthly_limit');

  for (const budget of budgets ?? []) {
    // 2. Calcular gasto do mês
    const { data: spent } = await supabase.rpc('get_category_spending', {
      p_user_id: budget.user_id,
      p_category: budget.category,
      p_month: currentMonth,
    });

    const pct = ((spent ?? 0) / budget.monthly_limit) * 100;

    // 3. Verificar e enviar alertas
    for (const threshold of [80, 100]) {
      if (pct < threshold) continue;

      const { data: existing } = await supabase
        .from('budget_alert_log')
        .select('id')
        .eq('budget_id', budget.id)
        .eq('threshold', threshold)
        .eq('month', currentMonth)
        .maybeSingle();

      if (existing) continue; // já enviado

      // Enviar push
      await fetch(`${Deno.env.get('NEXT_PUBLIC_APP_URL')}/api/push/send`, {
        method: 'POST',
        headers: {
          'Content-Type': 'application/json',
          Authorization: `Bearer ${Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')}`,
        },
        body: JSON.stringify({
          user_id: budget.user_id,
          title: threshold === 80 ? 'Orçamento em 80%' : 'Orçamento estourado!',
          body: threshold === 80
            ? `Você usou 80% do orçamento de ${budget.category} este mês`
            : `Orçamento de ${budget.category} estourado!`,
          url: '/more/orcamentos',
        }),
      });

      // Registrar no log
      await supabase.from('budget_alert_log').insert({
        user_id: budget.user_id,
        budget_id: budget.id,
        threshold,
        month: currentMonth,
      });
    }
  }

  return new Response(JSON.stringify({ ok: true }), { status: 200 });
});
```

### Função auxiliar para gasto por categoria (Supabase RPC)

Adicionar na migration `017`:

```sql
CREATE OR REPLACE FUNCTION get_category_spending(
  p_user_id uuid,
  p_category text,
  p_month text  -- formato YYYY-MM
) RETURNS numeric AS $$
  SELECT COALESCE(SUM(amount), 0)
  FROM transactions
  WHERE user_id = p_user_id
    AND category = p_category
    AND type = 'debit'
    AND to_char(date, 'YYYY-MM') = p_month;
$$ LANGUAGE sql SECURITY DEFINER;
```

### Deploy da Edge Function

```bash
supabase functions deploy check-budget-alerts --no-verify-jwt
```

`--no-verify-jwt` porque a função é chamada pelo pg_cron/cron interno, não por um usuário autenticado.

## Out of Scope

- Alertas por email
- Alertas por WhatsApp
- Configuração de threshold personalizado pelo usuário (os thresholds 80% e 100% são fixos)
- Alertas para metas financeiras (goals)
- Alertas para contas a vencer (scheduled_transactions) — essa é a feature "Bill Reminders"
- Desativar alertas por categoria individualmente
