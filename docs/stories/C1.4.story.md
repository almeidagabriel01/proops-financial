## Context

A feature de orçamentos já existe e funciona bem — `GET /api/budgets` calcula o gasto real vs limite para cada orçamento ativo. Mas o usuário só descobre que ultrapassou um limite se ele mesmo abrir o app e navegar até `/more/orcamentos`.

Todos os concorrentes sérios têm alertas de orçamento: Mobills, Organizze, YNAB, Monarch Money, PocketGuard. É a feature mais solicitada em reviews de usuários de apps financeiros: "perco o controle perto do final do mês".

Esta story adiciona a camada de notificação proativa: uma Supabase Edge Function que roda periodicamente, verifica os orçamentos de todos os usuários com push subscription ativa, e envia notificações nos thresholds de 80% e 100%. Uma tabela `budget_alert_log` evita spam de notificações (máximo uma notificação por threshold por orçamento por mês).

**Pré-requisito:** C1.3 (push subscriptions) deve estar implementada e deployada antes desta story.

---

## User Story

> Como usuário com orçamento configurado,
> quero receber uma notificação push quando atingir 80% e 100% do limite de uma categoria,
> para agir antes de estourar o orçamento
> sem precisar verificar o app manualmente.

## Tech Notes for @dev

### Rota de Cron (`/api/cron/check-budgets`)

```typescript
// src/app/api/cron/check-budgets/route.ts
export async function POST(request: Request) {
  // Vercel injeta automaticamente este header quando CRON_SECRET está configurado
  const authHeader = request.headers.get('authorization');
  if (authHeader !== `Bearer ${process.env.CRON_SECRET}`) {
    return Response.json({ error: 'Unauthorized' }, { status: 401 });
  }

  // Invocar a Edge Function com service role key
  const supabaseUrl = process.env.NEXT_PUBLIC_SUPABASE_URL!;
  const serviceRoleKey = process.env.SUPABASE_SERVICE_ROLE_KEY!;

  const response = await fetch(`${supabaseUrl}/functions/v1/check-budget-alerts`, {
    method: 'POST',
    headers: {
      Authorization: `Bearer ${serviceRoleKey}`,
      'Content-Type': 'application/json',
    },
  });

  if (!response.ok) {
    console.error('[cron/check-budgets] Edge Function error:', await response.text());
    return Response.json({ error: 'Edge Function failed' }, { status: 500 });
  }

  return Response.json({ ok: true, timestamp: new Date().toISOString() });
}
```

### Estrutura da Edge Function

```typescript
// supabase/functions/check-budget-alerts/index.ts
import { serve } from 'https://deno.land/std@0.177.0/http/server.ts';
import { createClient } from 'https://esm.sh/@supabase/supabase-js@2';

serve(async () => {
  const supabase = createClient(
    Deno.env.get('SUPABASE_URL')!,
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!,
  );

  const currentMonth = new Date().toISOString().slice(0, 7); // YYYY-MM

  // 1. Buscar todos os orçamentos
  const { data: budgets } = await supabase
    .from('budgets')
    .select('id, user_id, category, monthly_limit');

  for (const budget of budgets ?? []) {
    // 2. Calcular gasto do mês
    const { data: spent } = await supabase.rpc('get_category_spending', {
      p_user_id: budget.user_id,
      p_category: budget.category,
      p_month: currentMonth,
    });

    const pct = ((spent ?? 0) / budget.monthly_limit) * 100;

    // 3. Verificar e enviar alertas
    for (const threshold of [80, 100]) {
      if (pct < threshold) continue;

      const { data: existing } = await supabase
        .from('budget_alert_log')
        .select('id')
        .eq('budget_id', budget.id)
        .eq('threshold', threshold)
        .eq('month', currentMonth)
        .maybeSingle();

      if (existing) continue; // já enviado

      // Enviar push
      await fetch(`${Deno.env.get('NEXT_PUBLIC_APP_URL')}/api/push/send`, {
        method: 'POST',
        headers: {
          'Content-Type': 'application/json',
          Authorization: `Bearer ${Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')}`,
        },
        body: JSON.stringify({
          user_id: budget.user_id,
          title: threshold === 80 ? 'Orçamento em 80%' : 'Orçamento estourado!',
          body: threshold === 80
            ? `Você usou 80% do orçamento de ${budget.category} este mês`
            : `Orçamento de ${budget.category} estourado!`,
          url: '/more/orcamentos',
        }),
      });

      // Registrar no log
      await supabase.from('budget_alert_log').insert({
        user_id: budget.user_id,
        budget_id: budget.id,
        threshold,
        month: currentMonth,
      });
    }
  }

  return new Response(JSON.stringify({ ok: true }), { status: 200 });
});
```

### Função auxiliar para gasto por categoria (Supabase RPC)

Adicionar na migration `017`:

```sql
CREATE OR REPLACE FUNCTION get_category_spending(
  p_user_id uuid,
  p_category text,
  p_month text  -- formato YYYY-MM
) RETURNS numeric AS $$
  SELECT COALESCE(SUM(amount), 0)
  FROM transactions
  WHERE user_id = p_user_id
    AND category = p_category
    AND type = 'debit'
    AND to_char(date, 'YYYY-MM') = p_month;
$$ LANGUAGE sql SECURITY DEFINER;
```

`src: src/app/api/import/route.ts (Etapa 7.6)`

#### MAJOR

**M1 — `/api/push/check-budgets` sem especificação de autenticação**
`src: AC4`

A rota "interna" que dispara os alertas não tem AC de autenticação. Qualquer pessoa pode fazer `POST /api/push/check-budgets` e processar alertas de TODOS os usuários (custo de processamento + possível spam). A story C1.3 estabelece o padrão correto para `/api/push/send` (Bearer + SUPABASE_SERVICE_ROLE_KEY). O mesmo padrão deve ser exigido para `/api/push/check-budgets`.

Com Vercel Cron, a autenticação é feita via `CRON_SECRET` (Vercel injeta o header `Authorization: Bearer ${CRON_SECRET}`). AC4 deve especificar: validar `Authorization: Bearer ${CRON_SECRET}` e retornar 401 caso inválido.

#### MINORs

**mn1 — `get_category_spending` usa `to_char(date, 'YYYY-MM')` — impede uso de índice**
`to_char(date, 'YYYY-MM') = p_month` cria uma expressão que impede o uso do índice B-tree na coluna `date`. Para tabelas grandes, isso resulta em seq scan. Alternativa: `date >= (p_month || '-01')::date AND date < ((p_month || '-01')::date + interval '1 month')`. Para MVP é aceitável mas vale corrigir na migration.

**mn2 — TOCTOU: verificar log ANTES de enviar pode resultar em push duplo sob concorrência**
A Edge Function verifica `budget_alert_log` e depois insere. Se duas invocações rodarem em paralelo (improvável mas possível), ambas passam pela verificação antes que qualquer insert ocorra — resultado: push enviado duas vezes, depois o segundo insert falha na UNIQUE constraint (silenciosamente). Fix: inverter a ordem — tentar INSERT primeiro com `ON CONFLICT DO NOTHING`, checar se `count > 0` antes de enviar o push.
