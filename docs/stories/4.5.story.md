## Context

O produto tem 191 testes unitários passando — uma base sólida. Mas dois gaps críticos permanecem antes do lançamento:

**Gap 1 — Cobertura de backend:** NFR13 exige 80% de cobertura em `src/lib/`. A cobertura atual não está medida formalmente. Módulos como `billing/webhook-handler.ts` e `ai/categorizer.ts` têm testes, mas fluxos de integração (import → parse → categorize end-to-end) não estão cobertos.

**Gap 2 — Sem E2E:** Nenhum teste automatizado verifica que os fluxos críticos funcionam do ponto de vista do usuário real. Um refactor inocente em `/api/chat` ou no fluxo de login pode quebrar tudo silenciosamente.

**Gap 3 — CI/CD sem gates de qualidade:** O GitHub Actions (se existir) não bloqueia merges com testes falhando, cobertura abaixo de 80%, ou lint com erros.

Esta story fecha esses três gaps para que o lançamento em produção seja um evento de baixo risco.

---

## User Story

> Como desenvolvedor,
> quero que os fluxos críticos do produto sejam verificados automaticamente em cada push,
> e que um PR com testes falhando ou cobertura abaixo de 80% seja bloqueado automaticamente,
> para que eu possa lançar e iterar com confiança.

### CI — variáveis de ambiente para E2E (Supabase remoto de staging)

**Abordagem:** Opção A — banco remoto de staging, sem Docker. Os testes E2E apontam para um projeto Supabase de staging dedicado via secrets do GitHub Actions. Sem `supabase start`, sem Docker.

```yaml
# .github/workflows/ci.yml — job e2e
env:
  NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.TEST_SUPABASE_URL }}
  NEXT_PUBLIC_SUPABASE_ANON_KEY: ${{ secrets.TEST_SUPABASE_ANON_KEY }}
  SUPABASE_SERVICE_ROLE_KEY: ${{ secrets.TEST_SUPABASE_SERVICE_ROLE_KEY }}
```

Secrets a configurar no GitHub Actions (Settings → Secrets → Actions):
- `TEST_SUPABASE_URL`
- `TEST_SUPABASE_ANON_KEY`
- `TEST_SUPABASE_SERVICE_ROLE_KEY`

Para rodar E2E localmente, copiar essas vars para `.env.test.local`.

#### MAJORs (2)

**M1 — CI: `E2E_USER_EMAIL` não propagado entre steps — E2E silent skip no CI**

`src: .github/workflows/ci.yml` → job `e2e`, step "Create E2E test user"

O step cria o usuário via Node.js inline e tenta propagar o email com `::set-output` (sintaxe **depreciada e removida** dos runners Ubuntu recentes) e `console.log('E2E_USER_EMAIL=' + email)` (que apenas escreve no stdout, não define env var). O step seguinte `${{ env.E2E_USER_EMAIL }}` recebe string vazia → todos os 42 specs executam `test.skip()` → CI reporta **PASSED mas zero testes E2E rodaram**.

Fix obrigatório:
```yaml
# No step "Create E2E test user", substituir console.log por:
- name: Create E2E test user
  id: create-user
  run: |
    node -e "
    // ... (criar usuário)
    " >> $GITHUB_OUTPUT
    # ou via echo dentro do node:
    # process.stdout.write('user_email=' + email + '\n')
    # redirecionar para $GITHUB_OUTPUT via shell

# No step "Run E2E tests":
  E2E_USER_EMAIL: ${{ steps.create-user.outputs.user_email }}
```

Padrão correto (GitHub Actions): `echo "user_email=${EMAIL}" >> $GITHUB_OUTPUT` e referenciar via `${{ steps.create-user.outputs.user_email }}`.

#### MINORs (3)

**mn1 — Integration test "RLS check" usa service role — não verifica RLS real**

`tests/integration/import-flow.test.ts` → it('transações pertencem apenas ao usuário correto (RLS)'): a query usa o client inicializado com `TEST_SUPABASE_SERVICE_ROLE_KEY`, que bypassa RLS por definição.
