## Context

O produto tem 191 testes unitários passando — uma base sólida. Mas dois gaps críticos permanecem antes do lançamento:

**Gap 1 — Cobertura de backend:** NFR13 exige 80% de cobertura em `src/lib/`. A cobertura atual não está medida formalmente. Módulos como `billing/webhook-handler.ts` e `ai/categorizer.ts` têm testes, mas fluxos de integração (import → parse → categorize end-to-end) não estão cobertos.

**Gap 2 — Sem E2E:** Nenhum teste automatizado verifica que os fluxos críticos funcionam do ponto de vista do usuário real. Um refactor inocente em `/api/chat` ou no fluxo de login pode quebrar tudo silenciosamente.

**Gap 3 — CI/CD sem gates de qualidade:** O GitHub Actions (se existir) não bloqueia merges com testes falhando, cobertura abaixo de 80%, ou lint com erros.

Esta story fecha esses três gaps para que o lançamento em produção seja um evento de baixo risco.

---

## User Story

> Como desenvolvedor,
> quero que os fluxos críticos do produto sejam verificados automaticamente em cada push,
> e que um PR com testes falhando ou cobertura abaixo de 80% seja bloqueado automaticamente,
> para que eu possa lançar e iterar com confiança.

## Acceptance Criteria

### AC1: Cobertura de testes — atingir 80% em src/lib/

- [ ] Rodar `npm run test:coverage` e verificar a cobertura atual por módulo
- [ ] Identificar módulos em `src/lib/` com cobertura < 80% e adicionar testes faltantes
- [ ] Módulos prioritários a cobrir (se ainda não estiverem em 80%+):
  - `src/lib/ai/categorizer.ts` — lógica de categorização com cache e fallback
  - `src/lib/billing/plans.ts` — `getEffectiveTier`, limites por plano
  - `src/lib/billing/webhook-handler.ts` — todos os eventos Asaas
  - `src/lib/parsers/ofx-parser.ts` — casos edge (OFX malformado, FITID ausente)
  - `src/lib/parsers/csv-parser.ts` — todos os formatos de banco + deduplicação SHA-256
  - `src/lib/ai/chat.ts` — `buildFinancialContext`, `checkAndResetRateLimit`
- [ ] Configurar threshold em `vitest.config.ts`:
  ```typescript
  coverage: { thresholds: { lines: 80, functions: 80, branches: 80 } }
  ```
- [ ] `npm run test:coverage` falha (exit code 1) se cobertura < 80% em qualquer módulo de `src/lib/`
- [ ] **Verificação:** `npm run test:coverage` passa com todas as linhas >= 80% em src/lib/

### AC2: Testes de integração — fluxos críticos (Vitest)

- [ ] `tests/integration/import-flow.test.ts` — fluxo completo de importação:
  - Upload de arquivo CSV real (Nubank fixture)
  - Parsing e extração de transações
  - Categorização (mock do Claude API)
  - Salvamento no banco (Supabase remoto de staging via `TEST_SUPABASE_URL` + `TEST_SUPABASE_SERVICE_ROLE_KEY`)
  - Verificação que as transações aparecem na query do banco
  - Deduplicação: segunda importação do mesmo arquivo não cria duplicatas
- [ ] `tests/integration/webhook-flow.test.ts` — fluxo de pagamento:
  - POST no `/api/webhook/asaas` com payload de PAYMENT_CONFIRMED
  - Verificar que `subscriptions.status = 'active'` e `profiles.plan = 'pro'`
  - POST com SUBSCRIPTION_CANCELED → `profiles.plan = 'basic'`
- [ ] Testes de integração usam Supabase remoto de staging — sem Docker necessário. Documentar vars de ambiente necessárias
- [ ] Cada test run cria usuário de teste isolado (prefixo `test-{uuid}`) e faz cleanup no `afterAll`
- [ ] `npm run test:integration` script separado para não misturar com unit tests
- [ ] **Verificação:** `npm run test:integration` passa com vars `TEST_SUPABASE_*` configuradas

### AC3: Playwright — setup e configuração

- [ ] `npm run test:e2e` script configurado
- [ ] **Verificação:** `npm run test:e2e` roda sem erro (mesmo sem testes ainda)

### AC4: Playwright — testes E2E dos fluxos críticos

- [ ] `tests/e2e/auth.spec.ts` — autenticação:

- [ ] `tests/e2e/chat.spec.ts` — chat (modo stub):
  - Login com usuário Pro (trial) → acessar `/chat`
  - Enviar mensagem → resposta stub aparece (sem ANTHROPIC_API_KEY real no CI)

- [ ] Fixtures de usuário de teste E2E:

- [ ] **Verificação:** `npm run test:e2e` com app rodando → todos os 4 specs passam

### AC5: GitHub Actions — CI/CD com gates

- [ ] `.github/workflows/ci.yml` com jobs:

  ```yaml
  jobs:
    quality:
      # Roda em: todo push para main/dev, todo PR
      steps:
        - Checkout
        - Setup Node 20
        - npm ci
        - npm run lint         # Falha se lint errors
        - npm run typecheck    # Falha se type errors
        - npm run build        # Falha se build errors
        - npm run test         # Falha se unit tests falharem
        - npm run test:coverage # Falha se cobertura < 80%

    e2e:
      # Roda em: todo PR para main e pushes para main
      needs: [quality]  # só roda se quality passou
      steps:
        - Checkout
        - Setup Node 20
        - npm ci
        - npm run dev (background, com TEST_SUPABASE_URL apontando para staging)
        - npm run test:e2e
        - Upload screenshots on failure
  ```

- [ ] Secrets necessários no GitHub Actions para o job `e2e`:
  - `TEST_SUPABASE_URL` — URL do projeto Supabase de staging
  - `TEST_SUPABASE_ANON_KEY` — anon key do projeto de staging
  - `TEST_SUPABASE_SERVICE_ROLE_KEY` — service role do projeto de staging (para criar/deletar usuários de teste)

- [ ] Branch protection em `main`: requer `quality` job verde para merge
- [ ] Não precisa de secrets no CI para testes unit (usa mocks) — documentar quais secrets são necessários para E2E
- [ ] Cache de `node_modules` com `actions/cache` para builds mais rápidos
- [ ] **Verificação:** Abrir PR com teste falhando → CI bloqueia merge automaticamente

## Tech Notes for @dev

### Configuração de cobertura no Vitest

```typescript
// vitest.config.ts
export default defineConfig({
  test: {
    coverage: {
      provider: 'v8',
      include: ['src/lib/**'],
      exclude: ['src/lib/supabase/types.ts'], // gerado automaticamente
      thresholds: {
        lines: 80,
        functions: 80,
        branches: 80,
        statements: 80,
      },
      reporter: ['text', 'html', 'lcov'], // lcov para CI
    },
  },
});
```

### Playwright — fixture de usuário de teste

```typescript
// tests/e2e/fixtures/test-user.ts
import { createClient } from '@supabase/supabase-js';

const supabaseAdmin = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL!,
  process.env.SUPABASE_SERVICE_ROLE_KEY!,
);

export async function createTestUser() {
  const { data } = await supabaseAdmin.auth.admin.createUser({
    email: 'e2e-test@finansim.test',
    password: 'test-password-123',
    email_confirm: true,
  });
  return data.user;
}

export async function deleteTestUser(userId: string) {
  await supabaseAdmin.from('transactions').delete().eq('user_id', userId);
  await supabaseAdmin.auth.admin.deleteUser(userId);
}
```

### Playwright — login helper

```typescript
// tests/e2e/helpers/auth.ts
import type { Page } from '@playwright/test';

export async function loginAs(page: Page, email: string, password: string) {
  await page.goto('/login');
  await page.fill('[name="email"]', email);
  await page.fill('[name="password"]', password);
  await page.click('[type="submit"]');
  await page.waitForURL(/\/(dashboard|onboarding)/);

```

### CI — variáveis de ambiente para E2E (Supabase remoto de staging)

**Abordagem:** Opção A — banco remoto de staging, sem Docker. Os testes E2E apontam para um projeto Supabase de staging dedicado via secrets do GitHub Actions. Sem `supabase start`, sem Docker.

```yaml
# .github/workflows/ci.yml — job e2e
env:
  NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.TEST_SUPABASE_URL }}
  NEXT_PUBLIC_SUPABASE_ANON_KEY: ${{ secrets.TEST_SUPABASE_ANON_KEY }}
  SUPABASE_SERVICE_ROLE_KEY: ${{ secrets.TEST_SUPABASE_SERVICE_ROLE_KEY }}
```

Secrets a configurar no GitHub Actions (Settings → Secrets → Actions):
- `TEST_SUPABASE_URL`
- `TEST_SUPABASE_ANON_KEY`
- `TEST_SUPABASE_SERVICE_ROLE_KEY`

Para rodar E2E localmente, copiar essas vars para `.env.test.local`.

## QA Results

### Revisão Pré-Implementação — 2026-04-07
**Verdict:** CONCERNS

Um BLOCKER na especificação de E2E que precisa ser resolvido pelo @sm antes de implementar AC2/AC4/AC5.

#### BLOCKER

- **B1 — Playwright E2E requer Docker (Supabase local) — ambiente não disponível**: AC2, AC4 e AC5 especificam `supabase start` para testes de integração e E2E, o que requer Docker instalado localmente e no CI. O usuário confirmou que **não possui Docker instalado**. A story precisa ser revisada pelo @sm para adotar uma das alternativas:

  **Opção A (recomendada para MVP):** E2E rodando contra Supabase Cloud (projeto de teste dedicado) via variáveis de ambiente — sem Docker, funciona tanto localmente quanto no CI com secrets do GitHub Actions.

  **Opção B:** Reduzir escopo de E2E: testes Playwright usam MSW (Mock Service Worker) para interceptar chamadas Supabase — sem banco real, mas cobre fluxos de UI.

  **Opção C:** Remover AC2 (testes de integração com banco) do MVP e focar em AC1 (cobertura 80%), AC3+AC4 (E2E sem banco via Opção B), e AC5 (CI/CD com apenas o job `quality`).

  Esta decisão é de arquitetura de testes — requer alinhamento com o usuário antes de implementar.

#### Observações (sem impacto no gate)

- CI workflow `.github/workflows/ci.yml` já existe com jobs `quality` (lint + build + test) — story corretamente identifica que precisa ser expandido.
- Vitest coverage thresholds (`lines: 80, functions: 80, branches: 80`) estão bem configurados no tech note.
- Branch protection em `main` (AC5) altera o workflow atual onde pushes vão direto para `dev` — verificar impacto antes de ativar.

**NÃO autorizado para desenvolvimento até resolução do B1 pelo @sm.** ❌

B1 resolvido. @sm adotou Opção A (Supabase remoto de staging). AC2, AC4, AC5 e tech notes atualizados — sem dependência de Docker.
