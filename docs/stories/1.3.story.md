## Context

Esta e a story mais complexa do Epic 1. O fluxo de importacao e o core do produto — sem ele, nao ha dados para dashboard, categorizacao ou chat. O pipeline foi dividido em duas fases pelo QA review (B1 — BLOCKER):

- **Phase 1 (Sync):** Upload + parse + dedup + insert. Roda em API Route do Vercel com limite de 10s (Hobby tier). Nao inclui categorizacao.
- **Phase 2 (Async):** Categorizacao via Supabase Edge Function. Roda em Deno runtime (sem limite de tempo do Vercel). Notifica o frontend via Supabase Realtime quando completa.

Nesta story, implementamos apenas a **Phase 1 completa** + o **trigger para Phase 2** + a **UI de importacao com Realtime**. A logica de categorizacao real (IA) sera implementada no Epic 2, Story 2.1. Aqui, a Edge Function e um stub que marca todas as transacoes como `category='outros'`, `category_source='ai'` e muda o status para `completed`.

**QA Note m7:** Supabase Edge Functions rodam em **Deno runtime**, nao Node.js. Imports usam URLs ou `npm:` prefix, nao `require()`.
**QA Note m8:** Sequencia de status do import: `processing` → `categorizing` → `completed` (ou `failed`). O frontend deve observar essa progressao via Realtime.

### Supabase Edge Function (Deno)
```typescript
// supabase/functions/categorize-import/index.ts
// DENO RUNTIME — nao Node.js!

import { createClient } from 'npm:@supabase/supabase-js@2'

Deno.serve(async (req) => {
  const { importId, userId } = await req.json()
  
  const supabase = createClient(
    Deno.env.get('SUPABASE_URL')!,
    Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!
  )
  
  // Stub: marcar todas como 'outros' (categorizacao real no Epic 2)
  await supabase
    .from('transactions')
    .update({ category: 'outros', category_source: 'ai' })
    .eq('import_id', importId)
    .eq('category_source', 'pending')
  
  // Atualizar status do import
  await supabase
    .from('imports')
    .update({ status: 'completed' })
    .eq('id', importId)
  
  return new Response(JSON.stringify({ success: true }), {
    headers: { 'Content-Type': 'application/json' }
  })
})
```

### SHA-256 para CSV dedup
```typescript
// src/lib/parsers/utils.ts
import { createHash } from 'crypto'

export function normalizeDescription(raw: string): string {
  return raw
    .toLowerCase()
    .trim()
    .normalize('NFD')
    .replace(/[\u0300-\u036f]/g, '') // remove accents
    .replace(/[^a-z0-9 ]/g, '')     // remove special chars
    .replace(/\s+/g, ' ')           // collapse whitespace
}

export function generateExternalId(date: string, amount: number, description: string): string {
  const normalized = normalizeDescription(description)
  const input = `${date}|${amount}|${normalized}`
  return createHash('sha256').update(input).digest('hex')
}
```

### Supabase Realtime (client-side)
```typescript
// Na pagina de import, apos receber o importId:
const channel = supabase
  .channel(`import-${importId}`)
  .on('postgres_changes', {
    event: 'UPDATE',
    schema: 'public',
    table: 'imports',
    filter: `id=eq.${importId}`
  }, (payload) => {
    setImportStatus(payload.new.status)
  })
  .subscribe()

// Cleanup:
return () => { supabase.removeChannel(channel) }
```

### OFX Parser
- Considerar `ofx-js` (npm) ou parsing manual (OFX e SGML-like, nao XML)
- Formato tipico: `<STMTTRN>` blocks com `<DTPOSTED>`, `<TRNAMT>`, `<NAME>`, `<FITID>`
- FITID e o identificador unico da transacao (usado como external_id)

### CSV Bank Formats
- **Nubank:** `Data,Valor,Descrição` (sem header no mais recente, ou com header)
- **Itau:** separador `;`, formato de data `DD/MM/YYYY`, valor com virgula decimal
- **Bradesco:** similar ao Itau mas layout de colunas diferente
- Cada banco precisa de um adaptador que mapeia colunas para `ParsedTransaction`

### Supabase Storage RLS
- Path: `imports/{user_id}/{import_id}.{ext}`
- RLS ja configurada na Story 1.1: usuario so acessa arquivos no seu folder
