## Context

O app não possui sistema de tags. Usuários não conseguem agrupar transações por contexto transversal às categorias — por exemplo, todas as despesas de uma "viagem de negócios", ou marcar transações como "reembolsável" ou "presente".

Minhas Economias e Copilot Money/Monarch Money oferecem tags. No mercado BR, tags são uma feature diferenciada que poucos concorrentes têm. Junto com notas (C1.1), fecha o gap de "contextualização de transações" que usuários mais engajados precisam.

A UI deve ser inspirada no `CategorySelector` já existente, mas para tags: input com autocomplete das tags já usadas pelo usuário, chips removíveis, sem limite de tags por transação.

---

## User Story

> Como usuário,
> quero adicionar tags livres em minhas transações,
> para agrupar por contexto (ex: "viagem", "reembolsável", "presente")
> e filtrar transações por essas tags na página de histórico.

## Acceptance Criteria

### AC1: Migration — tabela `transaction_tags`

- [ ] Nova migration `015_transaction_tags.sql` em `supabase/migrations/`
- [ ] Schema da tabela:
  ```sql
  CREATE TABLE transaction_tags (
    id          uuid DEFAULT gen_random_uuid() PRIMARY KEY,
    transaction_id uuid NOT NULL REFERENCES transactions(id) ON DELETE CASCADE,
    user_id     uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
    tag         text NOT NULL CHECK (char_length(tag) BETWEEN 1 AND 50),
    created_at  timestamptz DEFAULT now()
  );
  ```
- [ ] Índices: `(user_id)`, `(transaction_id)`, `(user_id, tag)` para autocomplete eficiente
- [ ] Constraint única: `UNIQUE(transaction_id, tag)` — sem tags duplicadas na mesma transação
- [ ] RLS habilitada: `user_id = auth.uid()` para SELECT, INSERT, DELETE
- [ ] **Verificação:** `supabase db push` sem erros; INSERT com user_id errado retorna erro de RLS

### AC2: API — endpoints de tags

- [ ] `POST /api/transactions/[id]/tags` — adicionar tag a uma transação
  - Body: `{ "tag": "viagem" }`
  - Normalizar tag: trim + lowercase + sem espaços duplos
  - Se tag já existe: retornar 200 com a tag existente (idempotente)
  - Retorna: `{ id, transaction_id, tag, created_at }`
- [ ] `DELETE /api/transactions/[id]/tags/[tag]` — remover tag
  - Retorna 204 se removida, 404 se não encontrada
- [ ] `GET /api/tags/autocomplete?q=via` — buscar tags do usuário que começam com `q`
  - Retorna lista de strings únicas: `["viagem", "viagem-negócios"]`
  - Limite: 10 resultados
  - Se `q` omitido: retorna as 20 tags mais usadas pelo usuário (ordenadas por frequência)
- [ ] Todas as rotas verificam autenticação e RLS (user_id = usuário logado)
- [ ] **Verificação:** POST → tag salva; DELETE → tag removida; GET autocomplete → resultados corretos filtrados por user

### AC3: UI — input de tags na TransactionDetail sheet

- [ ] `src/components/transactions/transaction-detail.tsx` exibe seção "Tags" abaixo da seção de nota
- [ ] Chips das tags existentes exibidos com botão "×" para remover cada uma
- [ ] Input de texto para adicionar nova tag com:
  - Autocomplete usando `GET /api/tags/autocomplete` com debounce de 300ms
  - Sugestões exibidas como dropdown (shadcn Popover ou Command)
  - Pressionar Enter ou clicar na sugestão adiciona a tag imediatamente
  - Tags novas (não sugeridas) aceitas ao pressionar Enter
- [ ] Tag normalizada antes de salvar: trim + lowercase
- [ ] Limite visual: se > 10 tags, exibir "..." e número total
- [ ] Feedback visual de loading durante POST/DELETE
- [ ] Disponível para todos os usuários (Basic e Pro)
- [ ] **Verificação:** Adicionar tag "Viagem" → salva como "viagem", chip aparece; clicar "×" → tag removida; input mostra sugestões das tags existentes

### AC4: Filtro por tag na página /transactions

- [ ] `src/app/(app)/transactions/page.tsx` exibe filtro de tag no header de filtros
- [ ] Query param: `?tag=viagem` — persiste na URL ao navegar

## Tech Notes for @dev

### Migration

```sql
-- supabase/migrations/015_transaction_tags.sql
CREATE TABLE transaction_tags (
  id             uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  transaction_id uuid NOT NULL REFERENCES transactions(id) ON DELETE CASCADE,
  user_id        uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  tag            text NOT NULL CHECK (char_length(tag) BETWEEN 1 AND 50),
  created_at     timestamptz DEFAULT now(),
  CONSTRAINT transaction_tags_unique UNIQUE(transaction_id, tag)
);

CREATE INDEX idx_transaction_tags_user_id ON transaction_tags(user_id);
CREATE INDEX idx_transaction_tags_transaction_id ON transaction_tags(transaction_id);
CREATE INDEX idx_transaction_tags_user_tag ON transaction_tags(user_id, tag);

ALTER TABLE transaction_tags ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users manage own tags"
  ON transaction_tags FOR ALL
  USING (user_id = auth.uid())
  WITH CHECK (user_id = auth.uid());
```

### Revisão Pré-Implementação — 2026-04-10
**Agente:** Quinn (@qa)
**Verdict:** PASS ✅
**Gate:** 0 BLOCKERs · 0 MAJORs · 3 MINORs

**Autorizado para desenvolvimento.** ✅

#### MINORs

**mn1 — Autocomplete "top 20 por frequência" sem COUNT no tech note**
O tech note mostra `.ilike('tag', `${q}%`)` para buscas com `q`. Para o caso `q` omitido ("20 tags mais usadas"), a query precisa de `GROUP BY tag ORDER BY COUNT(*) DESC LIMIT 20`. O snippet atual não cobre esse caso — o @dev precisa implementar com aggregation separada.

**mn2 — Tech note mostra apenas inner join; chips na lista precisam de outer join**
O snippet de busca com filtro usa `transaction_tags!inner(tag)` (correto para filtrar). Para exibir chips em todas as transações sem filtro ativo, a query principal da página deve usar `select('*, transaction_tags(tag)')` (outer join). O @dev deve garantir que os chips apareçam mesmo quando nenhum filtro de tag está selecionado.

**mn3 — Sem feedback de erro para tag > 50 caracteres na UI**
O CHECK constraint do banco rejeitará com 400, mas não há AC para exibir mensagem amigável. Adicionar validação no frontend (antes do POST) e/ou mapear o erro 400 para toast "Tag muito longa (máximo 50 caracteres)".

**Agente:** Quinn (@qa)
**Verdict:** PASS ✅

**C1.5 autorizado para desenvolvimento.** ✅

#### Pontos críticos verificados

**RLS:** Correto. Três camadas independentes: política `FOR ALL USING (user_id = auth.uid()) WITH CHECK (user_id = auth.uid())` na migration + filtro `.eq('user_id', user.id)` explícito em cada API route. Sem vazamento entre usuários.

**Autocomplete — vazamento entre usuários:** Não ocorre. `src/app/api/tags/autocomplete/route.ts:21` aplica `.eq('user_id', user.id)` antes de qualquer retorno. Retorna exclusivamente tags do usuário autenticado.

**Filtro por tag com inner join:** Correto. `src/hooks/use-transactions.ts:44-46` usa `transaction_tags!inner(tag)` apenas quando `filters.tag` está ativo; outer join `transaction_tags(tag)` quando sem filtro, garantindo que chips apareçam em todas as transações.

**UI otimista e rollback:** Implementado nas duas direções (`handleAddTag` e `handleRemoveTag`). Rollback dispara `onTagsUpdated` para sincronizar o `TransactionItem` pai. Ver mn-pós-4 para edge case de concorrência.

**Limite de tags por transação:** Story declara "sem limite" explicitamente em Out of Scope. Implementação sem limite está correta por requisito.

#### MINORs pós-implementação

**mn-pós-1 — AC3: limite visual de 10 tags no TransactionDetail não implementado**
`transaction-detail.tsx:344-364` exibe todas as tags sem limite. AC3 especifica: "Limite visual: se > 10 tags, exibir '...' e número total". Usuários com muitas tags verão lista sem truncamento no sheet de detalhe. O limite de 3 chips + "+N" existe apenas no `TransactionItem` (AC5, correto). Risco: layout pode crescer indefinidamente em usuários com muitas tags.

**mn-pós-2 — AC3: ordem das seções Tags/Notas invertida**
`transaction-detail.tsx` renderiza Tags (linha 337) antes de Nota (linha 401). AC3 especifica "seção 'Tags' abaixo da seção de nota". Impacto visual minor — não afeta funcionalidade.

**mn-pós-3 — AC2: POST retorna `{ tag }` em vez de `{ id, transaction_id, tag, created_at }`**
`src/app/api/transactions/[id]/tags/route.ts:60` retorna `Response.json({ tag }, { status: 201 })`. AC2 especifica o shape completo: `{ id, transaction_id, tag, created_at }`. O frontend atual não consome `id` ou `created_at` do response (adiciona otimisticamente), então não há quebra funcional. Mas diverge do contrato da AC.

**mn-pós-4 — Rollback do add tag usa closure stale (concorrência)**
`transaction-detail.tsx:179-181`: `const rolledBack = tags.filter((t) => t !== tag)` usa o valor de `tags` capturado no fechamento da função, não o estado React atual. Se o usuário adicionar duas tags em rápida sucessão e a primeira falhar após a segunda já ter sido adicionada otimisticamente, o rollback remove a tag correta mas descarta a segunda adição. Cenário raro em produção, mas tecnicamente incorreto. Fix: usar `setTags(prev => prev.filter(t => t !== tag))` e `setTags(prev => [...prev, tag])` via functional update.

**mn-pós-5 — AC2: autocomplete com `q` usa limite 20 em vez de 10**
`src/app/api/tags/autocomplete/route.ts:38` usa `.slice(0, 20)` em todos os casos. AC2 especifica "Limite: 10 resultados" para busca com `q` e "20 tags mais usadas" para `q` omitido. A implementação retorna 20 em ambos os casos. Impacto mínimo (mais resultados que o especificado), mas diverge do AC.
