## Context

O app não possui sistema de tags. Usuários não conseguem agrupar transações por contexto transversal às categorias — por exemplo, todas as despesas de uma "viagem de negócios", ou marcar transações como "reembolsável" ou "presente".

Minhas Economias e Copilot Money/Monarch Money oferecem tags. No mercado BR, tags são uma feature diferenciada que poucos concorrentes têm. Junto com notas (C1.1), fecha o gap de "contextualização de transações" que usuários mais engajados precisam.

A UI deve ser inspirada no `CategorySelector` já existente, mas para tags: input com autocomplete das tags já usadas pelo usuário, chips removíveis, sem limite de tags por transação.

---

## User Story

> Como usuário,
> quero adicionar tags livres em minhas transações,
> para agrupar por contexto (ex: "viagem", "reembolsável", "presente")
> e filtrar transações por essas tags na página de histórico.

## Acceptance Criteria

### AC1: Migration — tabela `transaction_tags`

- [ ] Nova migration `015_transaction_tags.sql` em `supabase/migrations/`
- [ ] Schema da tabela:
  ```sql
  CREATE TABLE transaction_tags (
    id          uuid DEFAULT gen_random_uuid() PRIMARY KEY,
    transaction_id uuid NOT NULL REFERENCES transactions(id) ON DELETE CASCADE,
    user_id     uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
    tag         text NOT NULL CHECK (char_length(tag) BETWEEN 1 AND 50),
    created_at  timestamptz DEFAULT now()
  );
  ```
- [ ] Índices: `(user_id)`, `(transaction_id)`, `(user_id, tag)` para autocomplete eficiente
- [ ] Constraint única: `UNIQUE(transaction_id, tag)` — sem tags duplicadas na mesma transação
- [ ] RLS habilitada: `user_id = auth.uid()` para SELECT, INSERT, DELETE
- [ ] **Verificação:** `supabase db push` sem erros; INSERT com user_id errado retorna erro de RLS

### AC2: API — endpoints de tags

- [ ] `POST /api/transactions/[id]/tags` — adicionar tag a uma transação
  - Body: `{ "tag": "viagem" }`
  - Normalizar tag: trim + lowercase + sem espaços duplos
  - Se tag já existe: retornar 200 com a tag existente (idempotente)
  - Retorna: `{ id, transaction_id, tag, created_at }`
- [ ] `DELETE /api/transactions/[id]/tags/[tag]` — remover tag
  - Retorna 204 se removida, 404 se não encontrada
- [ ] `GET /api/tags/autocomplete?q=via` — buscar tags do usuário que começam com `q`
  - Retorna lista de strings únicas: `["viagem", "viagem-negócios"]`
  - Limite: 10 resultados
  - Se `q` omitido: retorna as 20 tags mais usadas pelo usuário (ordenadas por frequência)
- [ ] Todas as rotas verificam autenticação e RLS (user_id = usuário logado)
- [ ] **Verificação:** POST → tag salva; DELETE → tag removida; GET autocomplete → resultados corretos filtrados por user

### AC3: UI — input de tags na TransactionDetail sheet

- [ ] `src/components/transactions/transaction-detail.tsx` exibe seção "Tags" abaixo da seção de nota
- [ ] Chips das tags existentes exibidos com botão "×" para remover cada uma
- [ ] Input de texto para adicionar nova tag com:
  - Autocomplete usando `GET /api/tags/autocomplete` com debounce de 300ms
  - Sugestões exibidas como dropdown (shadcn Popover ou Command)
  - Pressionar Enter ou clicar na sugestão adiciona a tag imediatamente
  - Tags novas (não sugeridas) aceitas ao pressionar Enter
- [ ] Tag normalizada antes de salvar: trim + lowercase
- [ ] Limite visual: se > 10 tags, exibir "..." e número total
- [ ] Feedback visual de loading durante POST/DELETE
- [ ] Disponível para todos os usuários (Basic e Pro)
- [ ] **Verificação:** Adicionar tag "Viagem" → salva como "viagem", chip aparece; clicar "×" → tag removida; input mostra sugestões das tags existentes

### AC4: Filtro por tag na página /transactions

- [ ] `src/app/(app)/transactions/page.tsx` exibe filtro de tag no header de filtros
- [ ] Query param: `?tag=viagem` — persiste na URL ao navegar
- [ ] Quando tag selecionada: filtra transações que possuem essa tag (JOIN com `transaction_tags`)
- [ ] Exibir chips das tags de cada transação na `TransactionItem` (máximo 3, depois "+N")
- [ ] **Verificação:** Selecionar filtro "viagem" → apenas transações com essa tag exibidas; URL atualizada; chips visíveis na lista

### AC5: Exibição de tags na lista de transações

- [ ] `src/components/transactions/transaction-item.tsx` exibe chips de tags (max 3 visíveis + "+N")
- [ ] Chips com visual diferente das categorias (badge outline vs filled)
- [ ] **Verificação:** Transação com 2 tags → 2 chips visíveis; transação com 5 tags → 3 chips + "+2"

## Tech Notes for @dev

### Migration

```sql
-- supabase/migrations/015_transaction_tags.sql
CREATE TABLE transaction_tags (
  id             uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  transaction_id uuid NOT NULL REFERENCES transactions(id) ON DELETE CASCADE,
  user_id        uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
  tag            text NOT NULL CHECK (char_length(tag) BETWEEN 1 AND 50),
  created_at     timestamptz DEFAULT now(),
  CONSTRAINT transaction_tags_unique UNIQUE(transaction_id, tag)
);

CREATE INDEX idx_transaction_tags_user_id ON transaction_tags(user_id);
CREATE INDEX idx_transaction_tags_transaction_id ON transaction_tags(transaction_id);
CREATE INDEX idx_transaction_tags_user_tag ON transaction_tags(user_id, tag);

ALTER TABLE transaction_tags ENABLE ROW LEVEL SECURITY;

CREATE POLICY "Users manage own tags"
  ON transaction_tags FOR ALL
  USING (user_id = auth.uid())
  WITH CHECK (user_id = auth.uid());
```

## Out of Scope

- Tags hierárquicas (ex: "viagem/negócios")
- Limite máximo de tags por transação (sem limite por requisito)
- Tags compartilhadas entre usuários
- Cores customizadas por tag
- Tags automáticas sugeridas pela IA
- Busca full-text dentro das tags
- Bulk tagging (marcar múltiplas transações de uma vez)
