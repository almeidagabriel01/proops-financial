## Context

O SHA-256 já existe para deduplicação **na importação** — o `external_id` único por conta evita que a mesma transação do banco seja importada duas vezes. Mas isso não detecta quando o _usuário realmente foi cobrado duas vezes_ pelo mesmo serviço.

Rocket Money (internacional) e Copilot Money têm detecção de cobranças duplicadas como feature de destaque. É um dos itens mais elogiados em reviews: "economizei R$300/mês descobrindo cobranças indevidas".

O caso real: mesma assinatura cobrada duas vezes no mesmo mês, uma cobrança de supermercado duplicada (erro do comerciante), ou cobrança em dois cartões diferentes. A lógica é simples: mesma descrição normalizada + mesmo valor + datas próximas (janela de 3 dias).

O usuário pode dispensar o alerta marcando como "não é duplicata" — o que esconde permanentemente aquele par de transações dos alertas futuros.

---

## User Story

> Como usuário,
> quero ser alertado quando o app detectar possível cobrança duplicada,
> para identificar cobranças indevidas e agir rapidamente,
> sem precisar verificar manualmente cada transação.

## Tech Notes for @dev

### Normalização de descrição (consistente com SHA-256 existente)

```typescript
// src/lib/transactions/duplicate-detector.ts
function normalizeDescription(desc: string): string {
  return desc
    .toUpperCase()
    .replace(/[^A-Z0-9 ]/g, '')
    .replace(/\s+/g, ' ')
    .trim();
}
```

### Algoritmo de detecção (SQL direto no Supabase client)

```typescript
export async function detectDuplicates(
  supabase: SupabaseClient,
  userId: string,
  newTransactionIds: string[]
): Promise<number> {
  // Buscar as novas transações
  const { data: newTxns } = await supabase
    .from('transactions')
    .select('id, description, amount, type, date')
    .in('id', newTransactionIds)
    .eq('type', 'debit');

  let alertsCreated = 0;

  for (const txn of newTxns ?? []) {
    const normalized = normalizeDescription(txn.description);
    const dateObj = new Date(txn.date);
    const dateMinus3 = new Date(dateObj);
    dateMinus3.setDate(dateMinus3.getDate() - 3);
    const datePlus3 = new Date(dateObj);
    datePlus3.setDate(datePlus3.getDate() + 3);

    const { data: candidates } = await supabase
      .from('transactions')
      .select('id, description, amount, date')
      .eq('user_id', userId)
      .eq('type', 'debit')
      .eq('amount', txn.amount)
      .neq('id', txn.id)
      .gte('date', dateMinus3.toISOString().slice(0, 10))
      .lte('date', datePlus3.toISOString().slice(0, 10));

    for (const candidate of candidates ?? []) {
      if (normalizeDescription(candidate.description) !== normalized) continue;

      // Ordenar IDs para garantir unicidade do par
      const [id1, id2] = [txn.id, candidate.id].sort();

      const { error } = await supabase.from('duplicate_alerts').insert({
        user_id: userId,
        transaction_id_1: id1,
        transaction_id_2: id2,
        status: 'pending',
      });

      if (!error) alertsCreated++;
    }
  }

  return alertsCreated;
}
```

## Out of Scope

- Detecção em transações já existentes (histórico) — apenas nas novas importações
- Detecção de cobranças em valores ligeiramente diferentes (ex: R$10,00 vs R$10,01)
- Notificação push para duplicatas (pode ser adicionada em ciclos futuros)
- Cancelamento automático de cobranças duplicadas
- Detecção cross-account (entre contas bancárias diferentes do mesmo usuário)

**Autorizado para desenvolvimento.** ✅

A distinção entre SHA-256 (dedup de import) e detecção de duplicatas reais (cobrança indevida) está bem explicada no contexto — sem ambiguidade para o @dev.

#### MINORs

**mn1 — Janela de 3 dias: conservadora, mas correta para MVP**
Alguns bancos demoram até 5 dias para lançar débitos. Janela de ±3 dias pode perder alguns casos. Porém, alargar para ±5 dias aumenta os false positives (assinaturas mensais cobradas próximas, por exemplo). Manter 3 dias no MVP — documentar no Out of Scope como ajuste possível em ciclos futuros.

**mn2 — Ordering de IDs por convention sem CHECK CONSTRAINT na tabela**
AC1 diz "enforçar via trigger ou convention". O tech note usa apenas convention (`[txn.id, candidate.id].sort()`). Inserções diretas no banco (scripts de admin, testes) podem criar o par invertido `(B, A)` quando `(A, B)` já existe, passando pela UNIQUE constraint. Para maior robustez: adicionar `CHECK (transaction_id_1 < transaction_id_2)` na migration. Sem isso, a garantia é apenas de código — não de banco.

#### Verificações especiais

**SHA-256 vs detecção de duplicatas:** Distinção correta e preservada. `external_id` (SHA-256) continua como mecanismo de deduplicação na importação — evita reimportar a mesma transação do extrato. `duplicate_alerts` é um mecanismo de negócio diferente: detecta quando o usuário foi cobrado duas vezes pelo mesmo serviço. Sem sobreposição ou confusão nos dois sistemas. ✅

**Janela ±3 dias:** Implementada com UTC date math (`new Date(txn.date + 'T12:00:00Z')` + `setUTCDate`). Evita DST e ambiguidade de fuso corretamente. Filtro SQL via `.gte`/`.lte` aplicado antes do fetch, normalização de descrição aplicada em JS após. ✅

**Upsert idempotente:** `supabase.from('duplicate_alerts').upsert({ ... }, { onConflict: 'transaction_id_1,transaction_id_2', ignoreDuplicates: true })` — execuções repetidas para o mesmo par não criam registros duplicados nem erros. Testado em suite. ✅

**RLS:** Migration correta — `FOR ALL USING (user_id = auth.uid()) WITH CHECK (user_id = auth.uid())`. Fire-and-forget usa service-role client (correto para operação server-side), que seta `user_id` explicitamente. PATCH verifica ownership com `.eq('user_id', user.id)` + count=0 → 404. ✅

**mn2 pré-implementação resolvido:** CHECK CONSTRAINT `duplicate_alerts_ordered CHECK (transaction_id_1 < transaction_id_2)` adicionado na migration além da convention no código. Garantia no banco, não apenas no aplicativo. ✅

**Fire-and-forget:** IIFE `void (async () => { try { ... } catch { console.error } })()` lançada antes do `return Response.json()`, mas sem `await` — response retorna imediatamente enquanto detecção roda em background. Guard `if (newTransactions.length > 0)` evita disparo desnecessário. Falhas apenas logadas, não propagadas. ✅

**Card condicional:** `{duplicateAlerts.length > 0 && <DuplicateAlertsCard initialAlerts={duplicateAlerts} />}` no dashboard. Estado local no client component: quando todos os alertas são dispensados, `alerts.length` vai a zero e o componente retorna `null`. ✅

#### MINORs

**mn1 — Extra round-trip de DB no fire-and-forget**
A Etapa 7.5 re-busca IDs das transações por `import_id` (`getInvokeClient().from('transactions').select('id').eq('import_id', importId)`) em vez de usar o resultado do insert. A razão é técnica: o insert não usa `.select()`, então os IDs atribuídos pelo banco não estão disponíveis localmente. A re-busca é segura e mais robusta, mas adiciona 1 query extra por importação. Aceitável para MVP.

**mn2 — Limite inconsistente entre API GET e query do dashboard**
`GET /api/duplicate-alerts` retorna até 10 alertas; query do dashboard usa `.limit(5)`. Não causa inconsistência funcional — o dashboard faz query direta (server component), não chama a API. Mas a divergência pode confundir em testes futuros. Alinhar os dois para o mesmo valor em ciclo futuro.

**mn3 — `(supabase as any)` em duplicate_alerts nas API routes e dashboard**
Seguindo padrão existente de `scheduled_transactions`. Aceitável até a próxima regeneração de tipos com instância Supabase ativa. Documentado nas Completion Notes do @dev.
