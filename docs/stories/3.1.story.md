## Context

O schema de planos já existe (migration 003: `profiles.plan = 'basic' | 'pro'`, `audio_enabled`, `trial_ends_at`). `src/lib/billing/plans.ts` tem PLANS, PLAN_LIMITS e `getEffectiveTier()`. O que falta é a camada de UX: hook `use-plan.ts` para o cliente, paywall modal para upgrade, banner de trial, e enforcement do limite de 3 contas no Basic.

Esta story é pré-requisito para Story 3.2 (chat verifica plano via hook) e Story 3.4 (paywall precisa estar pronto antes do checkout).

**Decisão importante:** `use-plan.ts` é APENAS para UX (mostrar paywall, desabilitar botões). Toda verificação de autorização real é server-side na API route ou RLS. Nunca confiar apenas no frontend.

## User Story

> Como usuário Basic,
> quero ver claramente o que está incluído no meu plano e o que está disponível no Pro,
> para que eu entenda o valor do upgrade sem surpresas.

## Tech Notes for @dev

### Hook usePlan
```typescript
// src/hooks/use-plan.ts
'use client'
import { useUser } from '@/hooks/use-user'
import { getEffectiveTier } from '@/lib/billing/plans'

export interface PlanCapabilities {
  isBasic: boolean
  isPro: boolean
  inTrial: boolean
  trialDaysLeft: number
  canUseAudio: boolean
  canUseFunctionCalling: boolean
  aiModel: string
  aiMonthlyLimit: number
  maxBankAccounts: number
}

export function usePlan(): PlanCapabilities & { loading: boolean } {
  const { profile, loading } = useUser()
  if (!profile) return { ...defaults, loading }

  const tier = getEffectiveTier(profile.plan as 'basic' | 'pro', profile.trial_ends_at)
  const isPro = tier === 'pro'
  const inTrial = profile.plan === 'basic' && !!profile.trial_ends_at &&
    new Date(profile.trial_ends_at) > new Date()
  const trialDaysLeft = inTrial
    ? Math.ceil((new Date(profile.trial_ends_at!).getTime() - Date.now()) / 86400000)
    : 0

  return {
    isBasic: !isPro,
    isPro,
    inTrial,
    trialDaysLeft,
    canUseAudio: isPro && profile.audio_enabled,
    canUseFunctionCalling: isPro,
    aiModel: isPro ? 'claude-sonnet-4-6' : 'claude-haiku-4-5-20251001',
    aiMonthlyLimit: isPro ? 200 : 50,
    maxBankAccounts: isPro ? Infinity : 3,
    loading,
  }
}
```

### Issues

**M1 — Tech note de AC4 está incorreta: bank_accounts table JÁ EXISTE**
- **Severity:** MAJOR
- **Category:** requirements
- **Description:** A tech note de AC4 diz "A tabela `bank_accounts` pode não existir ainda" e propõe uma "Simplificação MVP" que exibe o paywall para TODOS os usuários Basic ao importar, independente da contagem real. Isso está errado — `bank_accounts` existe em `migration 001_initial_schema.sql` com `user_id` e `bank_name`. O @dev leria a tech note antes de verificar o schema e implementaria o comportamento incorreto: usuários Basic com 0 ou 1 conta veriam paywall desnecessário, quebrando o fluxo de primeiro import.
- **Correction:** Remover o parágrafo de "Simplificação MVP" da tech note. AC4 como escrito está correto — `supabase.from('bank_accounts').select('*', { count: 'exact', head: true }).eq('user_id', userId)` é exatamente o que deve ser feito.
