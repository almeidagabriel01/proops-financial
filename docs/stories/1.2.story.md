## Context

Com o projeto bootstrapped (Story 1.1), precisamos implementar o fluxo completo de autenticacao antes de qualquer feature de negocio. Toda a seguranca do app depende de Supabase Auth + RLS funcionando corretamente. Um usuario nao autenticado nao deve ver nada alem de login/signup.

Supabase Auth gerencia tudo: cadastro, login, sessao (JWT), refresh tokens, Google OAuth. O frontend usa `@supabase/ssr` para integrar com cookies do Next.js. O middleware do Next.js protege as rotas do grupo `(app)/`. RLS no banco garante isolamento de dados por usuario.

O trigger `handle_new_user()` (ja criado na Story 1.1 via migration) cria automaticamente um `profile` com `plan='free'` e `trial_ends_at = now() + 7 days` para cada novo usuario.

**PRD refs:** FR20 (cadastro email/Google), FR21 (sessao Supabase), FR22 (planos Free/Premium)
**Arch refs:** Auth Flow (section 6), RLS (section 3.2), Middleware (section 6.3), Profile trigger (section 3.2)

## Tech Notes for @dev

### Supabase Auth Setup
1. No Supabase Dashboard > Authentication > Providers:
   - Email: habilitar, ativar confirmacao de email
   - Google: habilitar, configurar com Google Cloud Console (OAuth credentials)
2. Redirect URL no Google Console: `https://<supabase-project>.supabase.co/auth/v1/callback`

### @supabase/ssr
- Instalar: `@supabase/ssr` (integra Supabase Auth com cookies do Next.js)
- Browser client: `createBrowserClient(url, anonKey)` — ja configurado na Story 1.1
- Server client para middleware: `createServerClient(url, anonKey, { cookies })` com cookie handling

### OAuth Callback Handler
```typescript
// src/app/(auth)/callback/route.ts
import { NextResponse } from 'next/server'
import { createServerClient } from '@supabase/ssr'

export async function GET(request: Request) {
  const { searchParams } = new URL(request.url)
  const code = searchParams.get('code')
  
  if (code) {
    const supabase = createServerClient(/* ... */)
    await supabase.auth.exchangeCodeForSession(code)
  }
  
  return NextResponse.redirect(new URL('/dashboard', request.url))
}
```

### Middleware Pattern
```typescript
// src/middleware.ts
// Usar createServerClient do @supabase/ssr
// Verificar session: supabase.auth.getUser() (NAO getSession — getSession nao valida JWT)
// IMPORTANTE: getUser() faz round-trip ao Supabase para validar; getSession() so decodifica localmente
```

### Layout do (app) group
- `src/app/(app)/layout.tsx` deve incluir auth guard (server-side via supabase server client)
- Se nao autenticado, redirect (redundancia com middleware para seguranca em camadas)
- Incluir shell basica: header com nome do app + botao logout
- Bottom nav sera adicionado na Story 1.4

### Mensagens de erro
- Supabase retorna erros em ingles (`Invalid login credentials`, `User already registered`, etc.)
- Mapear para PT-BR com um objeto de traducao simples:
```typescript
const AUTH_ERRORS: Record<string, string> = {
  'Invalid login credentials': 'Email ou senha incorretos',
  'User already registered': 'Este email ja esta em uso',
  // ...
}
```
