## Context

Epic 3 entregou planos, chat, function calling e áudio. O produto está funcional — mas o usuário que acabou de criar conta fica sozinho em frente a um dashboard vazio sem saber o que fazer.

Esta story fecha a experiência de primeiro uso: um fluxo guiado de 3-4 passos que conduz o novo usuário do cadastro até o primeiro dashboard com dados reais, em menos de 5 minutos.

**O problema real:** Exportar extrato bancário não é óbvio. Nubank esconde a opção em 3 taps. Itaú tem OFX somente no internet banking desktop. Sem instruções específicas por banco, o usuário abandona na primeira tela.

**O que esta story constrói:**
1. Fluxo de onboarding em steps (`/onboarding`) com stepper de progresso
2. Banco de instruções por banco (Nubank, Itaú, Bradesco, Santander, BB, Caixa)
3. Integração com o estado do usuário — onboarding marcado como completo no perfil
4. Mensagem de boas-vindas no primeiro login (se onboarding não completo)

**Dependências satisfeitas:** Story 1.2 (auth), Story 1.3 (import OFX/CSV), Story 3.1 (planos). Nenhuma story do Epic 4 como pré-requisito.

## User Story

> Como novo usuário,
> quero ser guiado passo a passo do cadastro até ver meu primeiro dashboard,
> para que eu entenda o valor do app em menos de 5 minutos sem precisar descobrir sozinho.

## Acceptance Criteria

### AC1: Fluxo de onboarding — estrutura

- [ ] Rota `/onboarding` com Client Component gerenciando steps
- [ ] 3 steps obrigatórios: `welcome` → `import` → `done`
- [ ] Stepper visual no topo: círculos numerados + linha de progresso
- [ ] Botão "Pular etapa" disponível em todos os steps exceto `done`
- [ ] Botão "Pular onboarding completo" no step `welcome` — redireciona direto ao dashboard
- [ ] Step `done` sem botão de pular — apenas "Ver meu Dashboard" → `/dashboard`
- [ ] **Verificação:** Navegar por todos os steps sem importar extrato → chega no dashboard

### AC2: Step 1 — Boas-vindas

- [ ] Saudação personalizada com o nome do usuário (de `profiles.full_name` ou email)
- [ ] Texto explicativo curto: o que o app faz e o que o usuário vai fazer agora
- [ ] Menção ao trial Pro ativo (7 dias) — CTA visual discreto informando o trial
- [ ] CTA principal: "Vamos começar" → avança para step 2
- [ ] Layout mobile-first: ilustração simples (SVG inline) + texto + botão
- [ ] **Verificação:** Nome do usuário aparece; trial mencionado se `trial_ends_at` no futuro

### AC3: Step 2 — Instruções de exportação por banco

- [ ] Lista de bancos suportados com instrução específica para cada um:

  | Banco | Formato | Instrução resumida |
  |-------|---------|-------------------|
  | Nubank | CSV | App → Perfil → Meu perfil → Exportar planilha |
  | Itaú | OFX | Internet banking desktop → Extrato → Exportar OFX |
  | Bradesco | CSV | App → Menu → Extrato → Exportar → CSV |
  | Santander | OFX | Internet banking → Conta → Extrato OFX |
  | Banco do Brasil | OFX | Internet banking → Extrato → Baixar OFX |
  | Caixa | CSV | App Caixa Tem → Extrato → Compartilhar como planilha |

- [ ] Cada banco: ícone/logo (placeholder texto bold OK no MVP), passos numerados, formato do arquivo
- [ ] Accordion ou tabs por banco — apenas um aberto por vez (mobile-friendly)
- [ ] Link para a tela de importação (`/import`) ou embed do componente de upload
- [ ] Ao importar com sucesso aqui: avança automaticamente para step 3
- [ ] Botão "Já importei" caso o upload foi feito antes de chegar aqui
- [ ] **Verificação:** Cada banco mostra instrução correta; importar arquivo → avança para step 3

### AC4: Step 3 — Conclusão

- [ ] Mensagem de sucesso se importação foi feita: "Suas transações estão sendo organizadas!"
- [ ] Mensagem alternativa se pulou importação: "Quando quiser, importe seu extrato na aba Importar"
- [ ] CTA único: "Ver meu Dashboard" → redireciona para `/dashboard`
- [ ] Confete ou animação de sucesso simples (CSS puro, sem lib externa)
- [ ] **Verificação:** Clicar "Ver meu Dashboard" → `/dashboard`

### AC5: Controle de estado do onboarding

- [ ] Coluna `onboarding_completed boolean DEFAULT false` adicionada à tabela `profiles` via migration
- [ ] Ao clicar "Ver meu Dashboard" no step 3: `PATCH profiles SET onboarding_completed = true`
- [ ] Ao pular onboarding completo no step 1: também marca como `onboarding_completed = true`
- [ ] Middleware ou lógica no `/(app)/layout.tsx`: se `onboarding_completed = false`, redirecionar para `/onboarding` na primeira visita ao dashboard
- [ ] Usuários existentes (com transações) com `onboarding_completed = false`: mostrar banner no dashboard em vez de redirecionar forçado
- [ ] **Verificação:** Novo usuário → `/onboarding`; usuário com onboarding completo → `/dashboard` direto

### AC6: Boas-vindas no primeiro login (usuário recorrente)

- [ ] Se usuário tem `onboarding_completed = false` mas chegou ao dashboard por outros meios: mostrar toast/banner de boas-vindas com CTA "Completar onboarding"
- [ ] Toast desaparece após 8 segundos ou ao clicar X
- [ ] Após onboarding completo: não mostrar novamente
- [ ] **Verificação:** Usuário com onboarding incompleto acessa `/dashboard` → vê banner com CTA

### AC7: Testes

- [ ] `tests/unit/onboarding/` — testar lógica de steps e redirecionamento
- [ ] Testar: novo usuário sem `onboarding_completed` → redirect para `/onboarding`

## Tech Notes for @dev

### Migration

```sql
-- supabase/migrations/008_onboarding.sql
ALTER TABLE public.profiles
  ADD COLUMN IF NOT EXISTS onboarding_completed boolean NOT NULL DEFAULT false;
```

**Usuários existentes:** `DEFAULT false` significa que todos os usuários existentes ficam com `onboarding_completed = false`. O banner no dashboard (AC5) trata isso sem redirecionar forçado — não quebra fluxo de usuários que já usam o app.

### Verificação no layout

```typescript
// src/app/(app)/layout.tsx — Server Component
// Verificar onboarding_completed e redirecionar se necessário
const { data: profile } = await supabase
  .from('profiles')
  .select('onboarding_completed, ai_queries_this_month')
  .eq('id', user.id)
  .single();

// Só redireciona se for rota de dashboard E usuário não tem transações ainda
// (evita redirecionar usuários existentes com dados)
const { count } = await supabase
  .from('transactions')
  .select('*', { count: 'exact', head: true })
  .eq('user_id', user.id);

if (!profile?.onboarding_completed && (count ?? 0) === 0 && pathname === '/dashboard') {
  redirect('/onboarding');
}
```

### Revisão Pós-Implementação — 2026-04-07
**Verdict:** CONCERNS

Zero BLOCKERs. Dois MAJORs devem ser corrigidos antes de marcar Done. Story 4.2 autorizada.

#### Respostas às perguntas específicas

- **AC5 — Redirect novo usuário**: Correto. Layout checa `!onboardingCompleted && !hasTransactions` → redireciona para `/onboarding`. `/onboarding` está fora do grupo `(app)` — sem loop. ✅
- **AC5 — Banner usuário existente**: Correto. `showOnboardingBanner = !onboardingCompleted && hasTransactions` — sem redirect forçado. ✅
- **AC3 — Instruções por banco**: Todos os 6 bancos presentes (Nubank CSV, Itaú OFX, Bradesco CSV, Santander OFX, BB OFX, Caixa CSV) com passos numerados e tips. ✅
- **AC4 — Avance automático**: Sim. `onImportSuccess()` é acionado pelo Realtime/polling quando `status === 'completed'` — chama `handleImportSuccess()` que faz `setStep(3)` sem ação do usuário. ✅
- **onboarding_completed = true quando**: Ao clicar "Ver meu Dashboard" no step 3 (`handleFinish`) OU "Pular onboarding" no step 1 (`handleSkipAll`). NÃO ao importar o extrato — a importação apenas avança para o step 3. Correto per AC5. ✅
- **Skip sem loop (condições normais)**: Funciona. O risco de loop está coberto no M2 abaixo.

#### MAJOR

- **M1 — `setTimeout` fora de `useEffect` em `OnboardingBanner`** (`src/components/layout/onboarding-banner.tsx:22`): `setTimeout(handleDismiss, 8000)` é chamado diretamente no corpo do componente. A cada re-render do layout, um novo timeout é criado e acumulado. Em React StrictMode (padrão no Next.js dev), o componente monta 2x — dois timeouts disparam, o banner desaparece antes dos 8s. Em produção, qualquer re-render parente (ex: `TrialBanner` muda estado) cria mais timeouts, causando dismiss prematuro e imprevisível.

  **Fix:**
  ```typescript
  useEffect(() => {
    const t = setTimeout(() => setDismissed(true), 8000);
    return () => clearTimeout(t);
  }, []);
  ```

- **M2 — `markOnboardingComplete()` sem error handling — loop de redirect potencial** (`src/app/onboarding/page.tsx:59-61, 64-67`): `handleSkipAll()` e `handleFinish()` (caminho sem importação) chamam `markOnboardingComplete()` e imediatamente fazem `router.push('/dashboard')` sem verificar se o update teve sucesso. Se o Supabase update falha silenciosamente (sem lançar exceção), `onboarding_completed` permanece `false` no banco. O layout detecta `!onboardingCompleted && !hasTransactions` e redireciona de volta para `/onboarding`. O usuário entra em loop até conseguir uma conexão estável.

  **Fix:**
  ```typescript
  async function markOnboardingComplete() {
    const { data: { user } } = await supabase.auth.getUser();
    if (!user) return false;
    const { error } = await supabase
      .from('profiles')
      .update({ onboarding_completed: true })
      .eq('id', user.id);
    return !error;
  }

  async function handleSkipAll() {
    const ok = await markOnboardingComplete();
    if (ok) router.push('/dashboard');
    // Se falhar: user permanece no onboarding — pode tentar de novo
  }
  ```

#### MINOR

- **m1 — `supabase` instanciado no render sem `useRef` em `page.tsx`** (`src/app/onboarding/page.tsx:18`): `const supabase = createClient()` é chamado a cada re-render, criando múltiplas instâncias. O mesmo dev usou `useRef` corretamente em `step-import.tsx`. Impacto baixo (o Supabase client reutiliza websocket internamente) mas é inconsistente. Fix: `const supabaseRef = useRef(createClient()); const supabase = supabaseRef.current`.

- **m2 — `<style>` inline em `step-done.tsx`** (`src/app/onboarding/_components/step-done.tsx:24-31`): CSS de animação `pop-in` injetado via tag `<style>` em JSX. Funciona no MVP mas pode causar FOUC em hidratação SSR e duplicação de regras em re-renders. Pode ser movido para `globals.css` em iteração futura — aceitável por ora.

**Story 4.2 autorizada para desenvolvimento.** ✅
