## Context

Nenhum concorrente brasileiro (nem internacional) implementa bem a detecção de sazonalidades fiscais e culturais do Brasil. É um "white space" de mercado identificado pela análise competitiva.

O contexto brasileiro tem gastos sazonais previsíveis e significativos: IPVA em janeiro, IPTU em fevereiro, material escolar em fevereiro, imposto de renda em março-abril, Black Friday em novembro, compras de natal em dezembro. Usuários são pegos de surpresa todos os anos — mesmo após o primeiro ano no app, não têm visibilidade de quanto gastaram naquele período.

A feature é contextual: aparece no mês relevante, usa o histórico do usuário para estimar o valor esperado, e orienta o usuário a se preparar. Sem nova tabela — lógica Server Component no dashboard.

---

## User Story

> Como usuário,
> quero ver alertas contextuais no dashboard sobre gastos sazonais previsíveis do calendário brasileiro,
> para me preparar financeiramente antes que cheguem IPVA, IPTU, material escolar e outras despesas sazonais,
> baseado no que eu mesmo gastei no mesmo período do ano anterior.

### Query para estimativa do ano anterior

```typescript
export async function getSeasonalityEstimate(
  supabase: SupabaseClient,
  userId: string,
  months: number[],
  keywords: string[]
): Promise<{ total: number; transactionCount: number } | null> {
  const lastYear = new Date().getFullYear() - 1;

  // Construir filtro de meses (ex: Janeiro do ano passado)
  const monthFilters = months.map(m =>
    `and(date.gte.${lastYear}-${String(m).padStart(2,'0')}-01,date.lt.${lastYear}-${String(m === 12 ? 1 : m+1).padStart(2,'0')}-01)`
  ).join(',');

  // Buscar transações com qualquer keyword nas descrições
  // Usar ilike com OR para múltiplas keywords
  let query = supabase
    .from('transactions')
    .select('amount')
    .eq('user_id', userId)
    .eq('type', 'debit');

  // Aplicar filtro de meses
  // Simplificação: buscar todas do ano passado nos meses corretos
  query = query
    .gte('date', `${lastYear}-01-01`)
    .lte('date', `${lastYear}-12-31`);

  const { data } = await query;

  // Filtrar client-side por keyword e mês (mais simples que SQL complexo)
  const matching = (data ?? []).filter(t => {
    const d = new Date(t.date);
    const txMonth = d.getMonth() + 1;
    if (!months.includes(txMonth)) return false;
    const desc = t.description.toLowerCase();
    return keywords.some(kw => desc.includes(kw.toLowerCase()));
  });

  if (matching.length === 0) return null;

  return {
    total: matching.reduce((sum, t) => sum + t.amount, 0),
    transactionCount: matching.length,
  };
}
```

> **Nota para @dev:** Se o volume de transações for alto (usuários com muitos dados), otimizar com uma query SQL direta usando `OR` de `ilike`. Para o MVP, a abordagem client-side é suficiente.

### Exibição condicional no dashboard


#### MINORs

**mn1 — `getSeasonalityEstimate` omite campo `yearMonth` do AC2**
AC2 especifica retorno `{ total: number, transactionCount: number, yearMonth: string }`. A implementação retorna `{ total: number, transactionCount: number }` — sem `yearMonth`. O campo não é consumido em nenhum lugar (card ou dashboard), portanto sem impacto funcional. Mas diverge do contrato definido no AC. Se `yearMonth` for necessário em ciclos futuros (ex: exibir "você gastou R$X em fevereiro/2025"), exigirá adição retroativa.

**mn2 — CTA link cobre apenas o mês atualmente visualizado em sazonalidades multi-mês**
Para IRPF (meses 3-4): ao visualizar abril 2026, `lastYearMonth = "2025-04"` e o CTA aponta para `/transactions?month=2025-04&search=irpf`. As transações de março/2025 (outro mês coberto pela mesma sazonalidade) ficam fora do link. Sem impacto funcional — o link é um ponto de partida para exploração, não um filtro exaustivo. Aceitável para MVP.

**mn3 — Keywords com acento não correspondem a descrições de extratos ASCII**
Extratos bancários brasileiros frequentemente usam ASCII sem acentuação por limitações históricas dos sistemas: "VEICULO CRLV SP", "MATRICULA ESCOLA", "FERIAS HOTEL". As keywords `'veículo'`, `'matrícula'`, `'férias'`, `'confraternização'` não vão corresponder a essas descrições via `includes()`. Cada sazonalidade afetada tem outras keywords sem acento que cobrem os casos mais comuns (ex: 'ipva', 'crlv', 'detran' para IPVA; 'hotel', 'viagem', 'passagem' para férias), então a feature funciona na maioria dos casos. Mas a cobertura é parcial. Fix: adicionar variantes sem acento às keyword lists ou implementar normalização de acento no `getSeasonalityEstimate`.

#### MAJOR

**M1 — Keyword `'black'` é excessivamente ampla e produz estimativas falsas**
`src: AC1 e Tech Notes — keywords do mês de Novembro`

A lista de keywords de novembro inclui `['black friday', 'blackfriday', 'black']`. O termo `'black'` sozinho vai corresponder a qualquer transação com "black" na descrição: `Starbucks Reserve Black`, `BlackBerry Device`, `Café The Black`, produtos da linha "Black" de qualquer marca, etc. Isso vai inflar a estimativa de Black Friday com gastos completamente não relacionados.

**Correção:** remover `'black'` da lista. As keywords `'black friday'` e `'blackfriday'` são suficientes e precisas para detectar compras reais de Black Friday.

#### MINORs

**mn1 — Inconsistência de tipo: `SeasonalityInfo.month` (singular) vs. `months: number[]` (plural)**
AC3 define `SeasonalityInfo` como `{ label: string, keywords: string[], icon: string, month: number }`. Mas o mapeamento `BRAZIL_SEASONALITIES` usa `months: number[]` (IRPF tem `[3, 4]`, férias tem `[6, 7]`). O tipo retornado por `getActiveSeasonalities` deve usar `months: number[]` ou re-exportar `SeasonalityConfig` — não criar um novo tipo com campo singular que conflita com o dado real.

**mn2 — "Zero impacto de performance" (AC5) é impreciso**
A função `getSeasonalityEstimate` busca TODAS as transações de um ano inteiro e filtra client-side. Para usuários com histórico extenso (1000+ transações/ano), isso traz um payload significativo. A afirmação "zero impacto de performance" em AC5 é incorreta. Corrigir para: "impacto baixo para volumes típicos de MVP; otimizar com query SQL direta se performance degradar em usuários com histórico denso".
